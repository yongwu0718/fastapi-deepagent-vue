# 任务属性评估与优先级排序引导

按以下阶段依次引导用户。阶段一用于 SKILL 步骤 4（预估未覆盖属性），阶段二到四用于 SKILL 步骤 5（构建周池+生成每日计划）。每完成一个阶段，确认用户无补充后再进入下一阶段。


## 阶段一：逐项属性评估

逐条读取 SKILL 步骤 3 级联拆解出的周任务和日行动列表，引导用户对每条填写以下四个维度。若用户不确定某项属性，使用维度末尾的提示问题辅助判断。

### 1.1 预计耗时

为每条行动选择耗时标签：

| 标签 | 适用场景 |
|------|---------|
| `<15分钟` | 单个简单操作：发邮件、填表、回消息 |
| `15-30分钟` | 短会议、审阅文档、写小段代码 |
| `30-60分钟` | 较长会议、写报告初稿、开发小功能 |
| `1-2小时` | 复杂任务单元：写完整方案、深度代码开发 |
| `2小时以上` | 需分解的大任务，标记为"**待分解**"并退回 SKILL 步骤 3 |

*提示：若用户的估计超过 2 小时，提醒该行动还需进一步分解。若用户某个维度反复犹豫，建议先用较短标签，记录为"保守估计"。*

### 1.2 精力类型

引导用户从以下四种类型中选择。若用户对分类不确定，读取 `energy_mapping.md` 的知识参考部分中的"四种精力类型定义"表格辅助判断。

- **🔵 深度专注**：需要高认知、持续注意力、创造性思考
- **🟡 沟通协作**：需要社交互动、实时反应、协调讨论
- **⚪ 机械执行**：重复性、低认知、流程化操作
- **🟢 学习充电**：理解吸收新信息，无即时产出压力

*提示：如果一项任务同时涉及两种类型，选择其"核心动作"对应的类型。例如"写PPT然后演示"——写是深度专注，演示是沟通协作，若两者不可拆分则按时间占比大的分类。*

### 1.3 依赖关系

为每条行动标注：

- **前置任务**：必须在我之前完成的事项（包括他人交付、审批、环境准备）。使用任务序号或简短描述标注。
- **阻塞点**：当前被谁/什么阻塞，预估解除时间。
- **可并行**：是否可与其它任务同时执行（勾选 ✓）。

*提示：若用户说"没有依赖"，追问"你不依赖任何人或任何前置条件就可以立即开始吗？"以二次确认。*

### 1.4 截止时间/窗口

记录每条行动的时效约束：

- **硬截止**：不可延迟的固定日期/时间。
- **软窗口**：理想完成时间范围，有弹性。
- **无约束**：无明确时间限制，纳入"重要不紧急"队列。

*提示：让用户区分"别人给我设的截止日期"和"我自己设定的理想日期"，避免自我施加过多硬性约束。*

## 阶段二：重要-紧急矩阵分类

完成属性评估后，引导用户将每条行动放入 2×2 矩阵。

### 引导规则

不要替用户直接判断，而是通过提问引导用户自己分类。对每条行动依次提问：

1. **紧急性判断**："这条任务如果今天/本周不做，会产生什么直接的负面后果？"
2. **重要性判断**："这条任务推进的是你长期目标的核心方向，还是仅仅解决当下的噪音？"

根据用户的回答，归入以下四个象限：

| 象限 | 处理策略 |
|------|---------|
| **紧急且重要** | → 安排在精力最佳时段，优先执行 |
| **重要不紧急** | → 计划入日程，设定明确的启动时间窗，防止滑入紧急象限 |
| **紧急不重要** | → 批量集中处理或考虑委派/简化为最小版本 |
| **不重要不紧急** | → 暂缓或用"改天清单"标记，不进入当前计划 |

### 检验清单

分类完成后逐项确认：
- 是否"紧急"任务过多？提醒用户：真正的紧急任务应该很少，多数是被拖延到最后一刻的"重要不紧急"。
- 是否有"不重要"任务占据了大量时间？建议降级或委派。

## 阶段三：依赖链拓扑排序

引导用户梳理任务间的前后关系，生成可执行的执行序列。

### 3.1 绘制依赖链

引导用户逐条确认：
- 哪些任务"零依赖"，可以立即开始？
- 哪些任务被阻塞，需要等待他人或其他任务完成？
- 阻塞链条最长的是哪条路径？

*提示：用序号和箭头画出简单依赖图，帮助用户可视化。例如：A → B → D，A → C。*

### 3.2 排序规则

按以下规则生成初始排序：

1. **先推出无前置任务**：所有依赖链最前端的任务优先进入待办队列。
2. **识别关键路径**：最长的依赖链是瓶颈路径，相关任务优先保障。
3. **标注阻塞点**：被他人阻塞的任务移入"等待队列"，设定定期检查时间。

### 3.3 输出

生成分组的任务序列：
- **可立即开始**：无前置依赖的任务列表
- **等待队列**：被阻塞的任务及阻塞原因
- **条件就绪**：前置任务即将完成的行动

## 阶段四：精力-时间块匹配

将阶段三的排序结果与用户精力曲线匹配，生成带时间窗的最终计划。

### 4.1 加载精力模型

读取 `energy_mapping.md` 阶段四输出的**个性化精力地图**，作为时间块分配基准。若用户尚未完成精力模式收集（energy_mapping 的阶段一到四），则回退使用 `energy_mapping.md` 中的标准双峰精力模型作为后备方案，并在输出中标注"使用默认精力模型（未经个性化校准）"。

### 4.2 匹配规则

按以下优先级将任务映射到时间块：

| 优先匹配 | 任务类型 | 推荐时间块 |
|---------|---------|-----------|
| 第一层 | 🔵 深度专注（紧急且重要） | 08:00-10:00 高峰 |
| 第二层 | 🔵 深度专注（重要不紧急） | 10:00-12:00 中高 |
| 第三层 | ⚪ 机械执行 | 13:00-15:00 午后低谷 |
| 第四层 | 🟡 沟通协作 | 15:00-17:00 恢复期 |
| 第五层 | 🟢 学习充电 / 弹性缓冲 | 17:00-19:00 |

### 4.3 例外处理

- **用户反馈精力模型不符**：根据用户实际反馈调整时间块，并在后续计划中应用调整后的模型。
- **高紧急任务挤占**：若当日紧急任务过多，优先确保最重要的深度任务获得至少 1 小时高峰期，其余按精力类型降级分配到其他时段。
- **连续深度任务**：连续深度专注不超过 3 小时，之间安排至少 30 分钟切换（推荐插入 15 分钟以内的机械任务作为"间隙填充"）。

### 4.4 输出

为每条行动标注最终的时间窗建议，格式：`[MM-DD | HH:MM-HH:MM] 任务描述`，完成后交由 SKILL 步骤 5 构建周池和每日计划。

## 对 Agent 的引导说明

- 每个阶段之间稍作停顿，确认用户完成后再推进。
- 属性评估阶段不评判用户估计是否合理，仅做记录和复述确认。
- 若用户对精力类型反复犹豫，默认建议偏向**较高级别**的精力需求，避免任务被安排在精力不足时段导致无法完成。
- 优先级矩阵阶段以提问为主，不直接给出分类结果，训练用户自身判断能力。
- 排序完成后，让用户整体浏览一次最终序列，询问是否有"直觉上感觉不对"的安排，这往往能暴露遗漏的隐含依赖或精力因素。
- 完成全部阶段后，立即转入 SKILL 步骤 5（构建周池+生成每日计划），不要在此阶段做额外分析。
