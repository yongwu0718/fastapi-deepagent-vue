# 待办事项清单规划

当用户请求规划待办事项清单时，严格遵循下方工作流。每完成一步，核对进度清单后再继续。

## 进度清单
- [ ] 步骤 1：收集任务（结构化解析 + 追问补全）
- [ ] 步骤 2：按时间跨度分类
- [ ] 步骤 3：级联拆解
- [ ] 步骤 4：预估未覆盖属性
- [ ] 步骤 5：构建周池 + 生成每日计划
- [ ] 步骤 6：确认并输出

## 首次配置（仅执行一次）

在首次使用本 skill 时，先完成以下配置，后续会话直接复用结果：

**精力地图收集**：加载 `references\energy_mapping.md`，按阶段一~四引导用户完成个性化精力-时间块匹配表。完成后标记为"已验证"，同会话中不再重复收集。后续步骤 4/5 直接引用已生成的精力地图。

若用户主动提出"精力节奏变了"，重新走一次 energy_mapping 即可。

### 步骤 1：收集任务

步骤 1 有两种模式，模式选择由用户输入自动判断：

| 模式 | 触发条件 | 行为 |
|------|---------|------|
| **增量收集**（默认） | 用户逐条贴入任务 | 模式匹配解析 → 缺口追问 → 输出结构化任务卡片 |
| **全量倾倒** | "做一次全面规划"、"帮我把所有事理出来" | 加载 `references\task_dump_prompt.md`，分阶段引导全量收集 |

#### 1.1 增量收集：模式匹配

对用户贴入的每条原始任务，按以下维度进行关键词匹配，自动预填属性：

**精力类型匹配**：

| 关键词/模式 | → 精力类型 |
|------------|-----------|
| 学、复习、看视频、看书、练、刷题、研究、阅读 | 🟢 学习充电 |
| 写、做、开发、设计、编程、画、分析、规划 | 🔵 深度专注 |
| 开会、沟通、汇报、演示、面试、对齐、评审 | 🟡 沟通协作 |
| 填、查、回复、整理、报销、归档、下载、更新 | ⚪ 机械执行 |

**时间跨度匹配**：

| 关键词/模式 | → 跨度 |
|------------|--------|
| "今天"、"下午"、"晚上"、无任何时间词且无长期暗示 | 🔵 日级 |
| "这周"、"下周"、"周三前"、明确周内表述 | 🟡 周级 |
| "下个月"、"月底前"、"1-2个月"、课程/项目类未指定长期 | 🟠 中期 |
| "今年"、"考研"、"年度"、"长期"、明确 >3 个月的目标 | 🟣 长期 |

**依赖关系匹配**：

| 关键词/模式 | → 依赖标注 |
|------------|-----------|
| "等...之后"、"需要...先"、"必须先...再" | 记录明确前置依赖 |
| "回复XX"、"审批"、"等XX交付" | 标记为阻塞（等待他人） |

**重要性/紧急性匹配**：

| 关键词/模式 | → 标注 |
|------------|--------|
| "必须"、"赶紧"、"deadline"、"最后期限"、带具体日期 | 紧急 |
| "重要"、"关键"、"核心" | 重要 |

#### 1.2 缺口追问

模式匹配完成后，逐条检查是否有**未填属性缺口**。按优先级追问：

1. **时间跨度未匹配** → "这项任务大概需要多久完成？今天/本周/这个月/更久？"
2. **精力类型不明确**（混合动作）→ "这个任务主要是思考创作，还是沟通配合？"
3. **缺少截止日期** → "有硬性截止日期吗？还是可以在本周内弹性安排？"
4. **微任务判断** → 满足 `<15分钟 + 单一动作 + 无依赖 + 当天` → 标记为随手池，不进入步骤 2

#### 1.3 输出结构化任务卡片

每条收集完成的任务输出为：

```
[任务名称] | 精力: 🟢 | 跨度: 🟣长期 | 截止: 12月 | 依赖: 无 | 重要性: 重要
```

收集完成后，将全部任务卡片清单交给步骤 2。

### 步骤 2：按时间跨度分类

将步骤 1 收集到的每个任务按时间跨度归入四个桶：

| 桶 | 跨度 | 典型示例 |
|----|------|---------|
| 🟣 长期 | >3 个月 | 考研、跳槽准备、年度目标 |
| 🟠 中期 | 1-3 个月 | 完成一个项目模块、学完一门课 |
| 🟡 周级 | 1-4 周 | 写报告、准备演示 |
| 🔵 日级 | ≤1 天 + 随手池 | 修 bug、回邮件、取快递 |

无明确截止日期的任务，引导用户给出一个季度内的目标位置，至少归入中期或长期。随手池任务直接跳过后续步骤，汇入每日计划的随手池区域。

### 步骤 3：级联拆解

按任务的时间跨度，自顶向下逐层拆解。每层拆出的子任务保留与该层的关联，方便后续弹性回退。

**拆解链路**：

```
长期 → 3月周期 → 月度里程碑 → 周任务 → 日行动
中期 →         月度里程碑 → 周任务 → 日行动
周级 →                      周任务 → 日行动
日级 →                               日行动
```

- **长期任务**：先按 3 个月一个周期拆分，每个周期拆为月度里程碑，每月再拆为周任务，每周拆为日行动。
- **中期任务**：跳过周期层，直接拆为月度里程碑 → 周任务 → 日行动。
- **周级任务**：跳过周期和月度层，直接拆为周任务 → 日行动。
- **日级任务**：直接作为日行动。

**日行动拆解规则**（单一动作 + 同一上下文）：

1. **模糊任务先澄清** — 无法描述"第一步做什么"的，标记为"待澄清"，追问产出定义后挂起。
2. **单一动作** — 一个行动只包含一个动词和一个对象（如"写完方案初稿"而非"搞定方案"），产出可验证。
3. **同一上下文** — 一个行动只在一个场景（工具/环境/地点）中完成，换场景即拆分。同上下文且均 <15 分钟的相邻行动可合并。
4. **超过 2 小时拆** — 预估超过 2 小时退回继续拆分；15 分钟以下的不再拆解。
   - **例外**：视频课、真题套卷等**认知连续性强的任务**可保留 2-3 小时的整块（如考研数学课 3 小时），但必须标注为"大队列任务"并在块内预留 1-2 次自主暂停点。
5. **拖延任务用最小锚点** — 习惯性拖延的任务，只拆出 5 分钟可完成的第一个动作，作为启动入口。

### 步骤 4：预估未覆盖属性

步骤 1 的模式匹配已预填了精力类型、时间跨度、依赖和重要性。本步骤仅处理步骤 1 未覆盖或标记为"待确认"的剩余属性。

加载 `references\task_attribute_prompt.md` 的阶段一，对步骤 3 拆出的每条周任务和日行动逐项补全：

- **预计耗时**：`<15分钟` / `15-30分钟` / `30-60分钟` / `1-2小时` / `2小时以上`。超过 2 小时的标记为"待分解"并退回步骤 3。
- **精力类型**：步骤 1 已匹配的直接确认，未匹配的追问用户。
- **依赖关系**：步骤 1 已记录的二次确认是否有遗漏。不得跳过二次确认。
- **截止时间/窗口**：步骤 1 未明确的，在此处补充：硬截止 / 软窗口 / 无约束。

### 步骤 5：构建周池 + 生成每日计划

这是整个工作流的核心。周任务是**唯一的弹性层**，每日计划是当天从周池拉取的快照。不存在"下周池"的概念。

#### 5.1 构建当前周池

加载 `references\task_attribute_prompt.md` 的阶段二（重要-紧急矩阵），对当前周所属的月度里程碑中所有周级任务进行重要性排序。

生成当前周池：

| 周池 | 来源 |
|------|------|
| 本周核心任务 | 硬截止在本周 / 阻塞链前端 / 紧急且重要 |
| 本周弹性任务 | 重要但可顺延 / 无硬截止 |

- **日计划时长按场景自适应**：
  | 用户类型 | 🔵 深度专注上限 | 日计划总量上限 | 典型场景 |
  |---------|---------------|--------------|---------|
  | 职场知识工作者 | 4 小时 | 4.8 小时 | 日常工作 + 学习 |
  | 全日制学习者（备考/学生） | 3.5 小时 | 6-7 小时 | 考研、考证、密集培训 |

- 时间跨度越长，日强度应越低。5 个月以上战线必须留足余量，避免前期透支。
- **每周休息节奏**：职场用户 ≥1 天弹性日；全日制用户必须安排 **1 天全休（不排任何学习）+ 1 天半休（≤2h 轻量回顾）**，长期战线不能只靠"不排满"。

#### 5.2 从周池拉取每日计划

每天早上从周池中按重要性降序拉取任务，结合 `references\energy_mapping.md` 的个性化精力地图分配时间块：

- 优先安排硬截止和阻塞链前端任务。
- 同优先级内按精力类型匹配时间块（深度→上午，沟通→下午，机械→午后低谷，学习→傍晚）。
- 依赖链未就绪的任务不拉入当天。

**强制休息规则（类比学校课间）**：
- 每个 ≥60 分钟的深度/学习块后，**必须插入 10-15 分钟物理休息**，不塞任何碎片任务。
- 连续深度专注不超过 90 分钟（含），超过即拆分为两块。
- 日计划表格中必须显式标注 ☕ 休息间隔，不允许任务首尾相连。

**晚间轮换规则**：
- 🔵 深度任务**不在同一天早晚各出现一次**。若上午已有高数刷题，晚间换为 🟢 学习充电或 ⚪ 机械复盘。
- 全日制备考用户：晚间高认知任务按天轮换（如周一三五高数错题，周二四六专业课笔记），避免同一科目每天双线疲劳。

**低效时段处理**：
- 用户明确标记为"刷手机"的低效时段，**不排正式学习任务**。
- 用"最小锚点"替代：只做 5 分钟即可完成的轻量动作（翻错题本/看 5 个单词/听 1 段音频），作为对抗拖延的启动入口，而非任务本身。

生成后按 `assets\daily_plan_schema.json` 结构保存为 JSON 文件（覆盖写入）。

**输出**：读取 `assets\daily_templates.md` 展示每日计划时间块，读取 `assets\quick_task_templates.md` 展示随手池。两者一并输出，并逐项检查各自模板中的验证规则表格，标注每项检查是否通过（✅/⚠）。

#### 5.3 日终弹性调整

每天结束时，根据完成情况动态调整周池：

| 完成情况 | 处理 |
|---------|------|
| ✅ 全部完成 | 无变化 |
| ⚠ 部分完成 | 剩余任务回到周池，**整池按重要性重排** |
| ❌ 全天跳过 | 当天所有任务回到周池，**整池按重要性重排** |

重排后，按重要性降序重新分配到本周剩余天数：

- **容量充足** → 全部保留，重新分配执行日期。
- **容量不足** → 从重要性最低的任务开始**删减**，被删的任务回到长期计划对应的月度里程碑中，不创建"下周池"。下周重新构建周池时，这些任务再次参与重要性排序。
- **容量富余** → 从当前月度里程碑中预拉下一个重要性最高的周任务进入本周池。

#### 5.4 运行时增量添加

周池运行中新增任务时，**走步骤 1 的增量收集流程**（模式匹配 + 缺口追问），然后：

**随手池（微任务）**：满足 `<15分钟 + 单一动作 + 无依赖 + 当天` → 丢入随手池，不占周池容量。

**标准通道（普通任务）**：走精简流程：

1. **级联拆解** — 仅对新增任务走步骤 3 的拆解链路。
2. **预估属性** — 补全步骤 1 未覆盖的属性。
3. **插入周池** — 加入当前周池，按重要性重新排序：

| 情况 | 处理 |
|------|------|
| 池容量充足 | 直接插入，标注 `[新增]` |
| 池容量不足 + 新任务更重要 | 删减池中重要性最低的任务（回月度里程碑），替换为新任务 |
| 池容量不足 + 新任务更不重要 | 新任务推迟到下周重新参与排位 |

- 新增任务若是长期/中期任务，同时更新长期计划中对应的月度里程碑。
- 若新任务导致池内弹性任务超 40%，自动降级低重要性任务回月度里程碑。

#### 5.5 长期计划输出

读取 `assets\lang_term_templates.md`，以表格形式向用户展示长期计划，包含时间轴、进度、截止预警。

连续 2 周同一任务被拉入周池但未完成 → 触发提醒：**"这个任务已连续 2 周未完成，降级重要性？进一步拆分？还是砍掉？"**

### 步骤 6：确认并输出

向用户展示最终计划，依次确认：
- 每个任务有明确的"完成"定义。
- 无超过 2 小时未分解的任务（视频课等"大队列任务"已标注暂停点的不在此限）。
- 高精力任务匹配高效能时段。
- 周池总量合理（弹性任务占比不超过 40%）。
- 连续顺延任务已触发提醒。
- 日强度与时间跨度匹配——如 5 个月以上战线，日深度 ≤3.5h、日总量 ≤7h，是否可持续。

根据反馈调整后，确认并输出最终计划。

## 弹性规则速查

| 触发条件 | 动作 |
|---------|------|
| 今天部分完成 | 剩余回周池 → 整池按重要性重排 |
| 今天全部未完成 | 全部回周池 → 整池按重要性重排 |
| 周池超支 | 按重要性从低到高删减 → 被删任务回到月度里程碑 |
| 周池富余 | 从月度里程碑预拉下一个高重要性任务 |
| 运行时新增任务 | 走步骤 1 模式匹配 → 微任务进随手池 / 普通任务进周池竞争排位 |
| 同一任务连续 2 周未完成 | 触发提醒：降级/拆分/砍掉 |
| 本周结束仍有未完成 | 回到月度里程碑，下周重建周池时重新参与排序 |

## 注意事项

- 不替用户判断重要性，通过提问引导用户自己决定。
- 跨项目任务需标注关联项目，避免重复。
- 每日高效专注时间和总量上限按 5.1 节场景表自适应（职场 4h/4.8h，全日制 3.5h/7h）。
- 不预建未来周池，每周周池在周初从长期计划中独立构建。
- 低效时段不排正式学习任务——用最小锚点替代，不把刷手机时间变成"第二战场"。
- 输出中不泄露任何用户未提及的个人信息或敏感上下文。
- `/health` 类简单检查不代表系统真正可用 —— 类比地，标记为"完成"的任务必须经过用户验证，不要仅凭自我报告判断。
