# LangSmith 数据平面

*数据平面* 由您的 Agent Server（部署）、其支持基础设施，以及持续轮询 LangSmith control plane 更新的“监听器”应用程序组成。

## 服务器基础设施

除了 Agent Server 本身，每个服务器的以下基础设施组件也包含在“数据平面”的广义定义中：

* **PostgreSQL**：用户、运行（run）和记忆数据的持久化层。
* **Redis**：工作进程（worker）的通信和临时元数据。
* **Secrets store**：环境密钥的安全管理。
* **Autoscalers**：根据负载对服务器容器进行自动扩缩容。

## “监听器”应用程序

数据平面“监听器”应用程序定期调用 control plane API 以：

* 确定是否应创建新的部署。
* 确定是否应更新现有部署（即新的修订版本）。
* 确定是否应删除现有部署。

换句话说，数据平面“监听器”读取 control plane 的最新状态（期望状态），并采取行动以协调未完成的部署（当前状态）使其匹配最新状态。

## PostgreSQL

PostgreSQL 存储服务器资源（threads、runs、assistants、crons）以及长期记忆存储中保存的项。它也是 checkpoint（graph 执行状态）的默认后端。您可以选择将 checkpoint 存储在 MongoDB 中——请参阅“配置 checkpointer 后端”。无论使用哪种 checkpointer 后端，PostgreSQL 始终是必需的。

## Redis

在每个 Agent Server 中，Redis 用于服务器与队列工作进程之间的通信，并存储临时元数据。Redis 中不存储任何用户或运行数据。

### 通信

Agent Server 中的所有运行都由每个部署中的后台工作进程池执行。为了支持这些运行的某些功能（例如取消和输出流式传输），我们需要一个用于服务器和处理特定运行的工作进程之间双向通信的通道。我们使用 Redis 来组织这种通信。

1. 一个 Redis 列表用作机制：一旦创建新运行，就唤醒一个工作进程。该列表中仅存储一个哨兵值，不存储实际的运行信息。工作进程随后从 PostgreSQL 中获取运行信息。
2. 结合使用 Redis 字符串和 Redis PubSub 通道，让服务器将运行取消请求传达给相应的工作进程。
3. 工作进程使用 Redis PubSub 通道在处理运行期间广播来自 agent 的流式输出。服务器中任何打开的 `/stream` 请求都将订阅该通道，并在事件到达时将其转发到响应中。任何时候都不会在 Redis 中存储事件。

### 临时元数据

Agent Server 中的运行可能会因特定失败而被重试（目前仅针对运行期间遇到的临时 PostgreSQL 错误）。为了限制重试次数（目前每个运行最多 3 次尝试），我们在运行被拾取时将其尝试次数记录在 Redis 字符串中。该字符串不包含除运行 ID 之外的任何特定运行信息，并在短暂延迟后过期。

## 数据平面特性

本节介绍数据平面的各项特性。

### 数据区域

**仅适用于 Cloud**
  数据区域仅适用于 Cloud 部署。

可以在 2 个数据区域中创建部署：US 和 EU

部署的数据区域由创建该部署的 LangSmith 组织的数据区域隐含决定。部署及其底层数据库不能在数据区域之间迁移。

### 自动扩缩容

`Production` 类型的部署会自动扩展到最多 10 个容器。扩缩容基于 3 个指标：

1. CPU 利用率
2. 内存利用率
3. 待处理运行（进行中的运行）数量

对于 CPU 利用率，自动扩缩容器的目标利用率为 75%。这意味着自动扩缩器将增加或减少容器数量，以确保 CPU 利用率处于或接近 75%。对于内存利用率，自动扩缩器的目标利用率也是 75%。

对于待处理运行数量，自动扩缩器的目标是 10 个待处理运行。例如，如果当前容器数量为 1，但待处理运行数量为 20，则自动扩缩器会将部署扩容到 2 个容器（20 个待处理运行 / 2 个容器 = 每容器 10 个待处理运行）。

每个指标独立计算，自动扩缩器将根据导致最大容器数量的指标来确定扩缩容操作。

并非所有指标都适用于每种容器类型。队列工作进程根据待处理运行数量进行扩缩容——当积压增加时，会启动更多工作进程来消耗积压。API 服务器根据 CPU 和内存进行扩缩容，以响应客户端请求量。这意味着运行提交的激增不会减慢读取操作（例如获取线程状态）。关于自托管的配置详情，请参阅“为扩缩容配置 Agent Server”。

缩容操作在执行任何操作之前会延迟 30 分钟。换句话说，如果自动扩缩器决定缩容某个部署，它将首先等待 30 分钟再执行缩容。30 分钟后，重新计算指标，如果重新计算的指标导致容器数量少于当前数量，则部署将缩容。否则，部署保持扩容状态。这个“冷却”期确保部署不会过于频繁地扩缩容。

### 静态 IP 地址

**仅适用于 Cloud**
  静态 IP 地址仅适用于 Cloud 部署。

2025 年 1 月 6 日之后创建的部署的所有流量都将通过一个 NAT 网关。此 NAT 网关根据数据区域具有多个静态 IP 地址。有关静态 IP 地址列表，请参阅“Allowlist IP addresses”表。

### 负载大小

**仅适用于 Cloud**
  负载大小限制仅适用于 Cloud 部署。

发送到 Cloud 部署的所有请求的最大负载大小为 25 MB。尝试发送负载超过 25 MB 的请求将导致 `413 Payload Too Large` 错误。

### 自定义 PostgreSQL

自定义 PostgreSQL 实例仅适用于自托管部署。

可以使用自定义 PostgreSQL 实例替代由 control plane 自动创建的实例。指定 `POSTGRES_URI_CUSTOM` 环境变量以使用自定义 PostgreSQL 实例。

多个部署可以共享同一个 PostgreSQL 实例。例如，对于 `Deployment A`，可以将 `POSTGRES_URI_CUSTOM` 设置为 `postgres://:@/?host=`；对于 `Deployment B`，可以将 `POSTGRES_URI_CUSTOM` 设置为 `postgres://:@/?host=`。`` 和 `database_name_2` 是同一实例中的不同数据库，但 `` 是共享的。**同一个数据库不能用于不同的部署**。

### 自定义 Redis

自定义 Redis 实例仅适用于自托管部署。

可以使用自定义 Redis 实例替代由 control plane 自动创建的实例。指定 `REDIS_URI_CUSTOM` 环境变量以使用自定义 Redis 实例。

多个部署可以共享同一个 Redis 实例。例如，对于 `Deployment A`，可以将 `REDIS_URI_CUSTOM` 设置为 `redis://:/1`；对于 `Deployment B`，可以将 `REDIS_URI_CUSTOM` 设置为 `redis://:/2`。`1` 和 `2` 是同一实例中的不同数据库编号，但 `` 是共享的。**同一个数据库编号不能用于不同的部署**。

### MongoDB checkpointing

适用于 Cloud（使用外部管理的 MongoDB 实例）和 Standalone 部署。

您可以使用 MongoDB 作为 checkpoint 存储的替代后端。配置后，MongoDB 仅处理 checkpoint 数据——所有其他服务器资源仍然需要 PostgreSQL。

设置说明请参阅“配置 checkpointer 后端”。
