# Deep Agents 框架全景

Deep Agents 是一个模块化的长期代理框架。它并非单一模型调用的包装，而是通过将规划、文件系统、上下文管理、子代理委派、代码执行、人机协同等能力融合进一个可组合的运行时，使代理能够胜任跨越多个对话轮次、甚至多个会话的复杂任务。理解这一框架的关键在于把握两层关系：**每一能力提供了什么**，以及**这些能力如何通过上下文工程、虚拟文件系统和配置文件织成一张协同网络**。

## 规划能力

框架内置了 `write_todos` 工具，让代理可以像人类一样维护一份结构化的待办清单。任务状态在 `pending`、`in_progress` 和 `completed` 之间流转，并持久保存在代理状态中。这不是简单的列表，而是代理对多步骤工作进行分解、跟踪和调整的核心手段——尤其适合需要长时间规划的任务。它帮助代理在上下文窗口变化时仍不丢失目标，是长期运行的第一块基石。

> `write_todos` 工具详细 API 参考见 [TodoListMiddleware](todomiddleware.md)

## 虚拟文件系统与后端

代理不能只靠对话历史工作。Deep Agents 提供了一个统一的虚拟文件系统，将 `ls`、`read_file`、`write_file`、`edit_file`、`glob`、`grep` 等工具暴露给模型。这些工具不直接操作宿主机磁盘，而是通过可插拔的**后端**完成。后端决定了文件存储在哪里、持久化范围以及安全边界。

框架内置了多种后端：
- **StateBackend**（默认）将文件保存在当前对话线程的 LangGraph 状态中，适合临时工作区。
- **FilesystemBackend** 直接读写本地磁盘，允许代理访问指定根目录下的真实文件，需谨慎开启 `virtual_mode` 来限制路径遍历。
- **LocalShellBackend** 在 FilesystemBackend 之上额外提供 `execute` 工具，让代理直接在主机上运行 Shell 命令。这赋予了巨大能力，但安全风险极高，仅限个人开发环境。
- **StoreBackend** 将文件存入 LangGraph Store 实现跨会话持久化，是实现长期记忆的基础。
- **ContextHubBackend** 利用 LangSmith Hub 仓库持久化文件，附带提交历史。
- **CompositeBackend** 是路由器，能够将不同路径前缀（如 `/memories/`、`/workspace/`）映射到不同的后端，在同一代理中同时拥有线程作用域的临时文件和跨会话的持久存储。

所有后端都遵循 `BackendProtocol`，保证工具层无需感知切换。其中，沙箱后端（如 Modal、Daytona）不仅提供文件系统，还提供 `execute` 工具，在隔离环境中运行任意命令，是代码执行的安全基础。

> 虚拟文件系统和后端 详细内容请参考见 [Backends](index/langchain-index/deepagent/concepts/backends.md)
## 文件系统权限

在虚拟文件系统之上，框架支持声明式权限规则。通过 `permissions` 参数传入 `FilesystemPermission` 列表，可以精确控制代理对哪些路径拥有读或写权限。规则按声明顺序“先匹配优先”求值，无匹配则默认允许。这让你可以将代理限制在工作目录内，保护 `.env` 等敏感文件，或为子代理赋予比父代理更窄的访问边界。

需要注意的是，权限**不适用于沙箱后端**——因为沙箱内通过 `execute` 可执行任意命令，路径限制形同虚设。在那类场景下，你需要依赖网络阻断、策略钩子和人机协同来构建安全防线。

> 文件系统权限 详细内容请参考见 [Permissions](index/langchain-index/deepagent/concepts/permissions.md)

## 任务委派：子代理

当代理需要执行会产生大量中间输出（如网络搜索、长文件读取）的任务时，主代理的上下文窗口会迅速被填满。子代理正是为解决这种**上下文膨胀**而设计的。

主代理通过 `task` 工具创建子代理——一个拥有独立上下文、独立工具集甚至独立模型的代理实例。子代理自主执行任务，最终只将一份报告返回给主代理，中间步骤全部隔离。这带来了几个重要优势：
- **上下文隔离**：主代理只看到最终结果，保持对话精炼。
- **并行执行**：多个子代理可以并发运行。
- **专业化分工**：不同的子代理可以配置不同的工具、技能、模型和权限。
- **令牌效率**：压缩大型子任务的上下文，仅消耗一份报告的空间。

子代理分为两种执行模式。**同步子代理**会让主代理阻塞等待结果，适合需要立即使用结果的场景。每个同步子代理都有独立的 `interrupt_on`、`permissions`、`skills` 等配置，并可通过 `response_format` 要求结构化输出，使父代理获得可解析的 JSON。

**异步子代理**则将任务作为后台进程启动，主代理立即拿到任务 ID 并可继续与用户交互。监督者随后可以使用 `start_async_task`、`check_async_task`、`update_async_task`、`cancel_async_task` 和 `list_async_tasks` 等工具来管理后台任务。异步子代理通过代理协议服务器（通常为 LangGraph 部署）通信，支持 ASGI 进程内传输或 HTTP 远程传输，既可以共同部署，也可以拆分到不同服务器上。

子代理的存在也让流式传输变得层次化：你可以从主代理和每个子代理独立地流式传输消息、工具调用和状态，构建出多面板的实时观察界面。

> 子代理 详细内容请参考见 [Subagent](index/langchain-index/deepagent/concepts/subagent.md)
> 异步子代理 详细内容请参考见 [Async Subagent](async_subagent.md)
## 上下文工程：编排一切

如果虚拟文件系统是基础设施，那么上下文工程就是将这些资源有序注入代理循环的编排层。上下文工程解决一个根本问题：**在正确的时间以正确的格式将正确的信息交给模型**。

整个上下文系统划分为三种类型：
- **模型上下文（瞬态）**：控制每次模型调用的输入，包括系统提示、消息历史、可用工具、模型选择和响应格式。通过中间件，这些都可以动态调整——例如根据用户角色决定工具集，或根据对话长度切换模型。
- **工具上下文（持久读写）**：工具不仅是执行器，还能通过 `ToolRuntime` 读取静态配置（如 API 密钥）、代理状态（State）和长期记忆（Store），并将结果写回 State 或 Store，形成持久的记忆累积。
- **生命周期上下文（持久）**：发生在模型调用和工具执行之间的动作，例如通过摘要中间件将过长历史压缩并持久化到文件系统，或添加日志和护栏。

所有这一切的核心是 **Runtime**——一个依赖注入的统一入口。它封装了静态配置（Context）、长期存储（Store）、流式输出器、执行元信息和服务器信息。无论是在工具还是中间件中，你都可以通过注入的 `Runtime` 或 `ToolRuntime` 安全地访问这些资源，避免硬编码。

上下文压缩是维持代理长期运行的关键。当工具调用输入或结果过大时，框架会自动**卸载**内容到虚拟文件系统，仅保留文件引用和预览。当对话历史接近模型令牌限制时，旧消息会被 LLM 生成的结构化摘要**替换**，完整的原历史则存入文件系统备查。这种“摘要 + 文件归档”的双重机制让代理在不丢失细节的前提下持续运作。

> 上下文压缩 详细内容请参考见 [Context Engineering](index/langchain-index/deepagent/concepts/context_engineering.md)

## 代码执行：沙箱与解释器

代理常常需要运行代码。Deep Agents 提供了两条互补路径：

**沙箱后端**在隔离环境（Modal、Daytona、Deno 等）中运行。代理获得标准文件系统工具外加 `execute` 工具，可以执行任意 Shell 命令、安装依赖、运行测试。沙箱是你与代理之间的安全边界：它保护主机不受代理文件操作和命令执行的影响，但不防护上下文注入攻击——攻击者仍可在沙箱内部运行任意命令并通过网络泄露数据。因此，机密信息绝不应放入沙箱，敏感操作应配合人机协同。

**解释器**基于 QuickJS 运行时，提供一种更轻量的代码执行方式。代理通过 `eval` 工具编写 TypeScript 程序，在内存中执行逻辑，捕获 `console.log` 并返回最后一个表达式的值。解释器的核心价值在于**编程式工具调用**：你可以将选定的工具（如 `task`）暴露为 JavaScript 函数，代理在代码中循环、并发、条件调用它们，仅将最终的综合结果返回模型，中间状态保留在运行时变量中。这种模式对于批处理、递归子代理调用、数据转换和探索大变量空间极其高效。

解释器还支持**技能模块**——将经过测试的辅助函数（如解析器、评分器）作为可导入模块暴露给解释器，代理无需让模型重新生成这些确定性的代码。解释器状态默认会在对话回合间快照持久化，结合 LangGraph 检查点器甚至可以实现“时间旅行”调试。

> 沙箱 详细内容请参考见 [Sandbox](sandboxs.md)
> 解释器 详细内容请参考见 [Interpreter](index/langchain-index/deepagent/concepts/Interpreters.md)

## 人机协同

对于高风险操作，框架可以在指定的工具调用处暂停执行，等待人工审批。通过 `interrupt_on` 参数，你可以为每个工具配置允许的决策类型：批准、编辑参数、拒绝或直接回复。当代理调用这些工具时，执行中断，返回控制权给应用程序，收集人工决策后通过 `Command` 恢复。

中断机制也适用于子代理内部，甚至可以与 `interrupt()` 原语结合，在自定义工具中动态请求审批。这为破坏性操作、昂贵 API 调用和交互式调试提供了安全闸门。

> 人机协同 详细内容请参考见 [Human-in-the-loop](index/langchain-index/deepagent/concepts/Human-in-the-loop.md)

## 技能与记忆：输入的两种形态

**记忆**是始终加载的持久上下文。通过 `memory` 参数指定的 `AGENTS.md` 文件，代理在每次启动时都会将其内容注入系统提示，用于存储编码风格、项目惯例、用户偏好等。记忆可以按作用域隔离：代理作用域（所有用户共享）、用户作用域（每个用户独立）或组织作用域（全局策略，通常只读）。代理可以使用 `edit_file` 工具更新记忆，也可以选择通过后台合并代理在对话之间异步处理记忆更新，以减少热路径延迟。

**技能**则采用渐进式披露。每个技能是一个包含 `SKILL.md` 的目录，其前置元数据中的名称和描述在启动时被读取，但完整内容直到代理判定相关时才被加载。这让技能能够承载详细的工作流指令、参考文档和脚本，而不占用常驻上下文。技能还可以包含解释器模块（向 QuickJS 暴露可导入函数），或在沙箱中执行的脚本。技能与记忆相辅相成：记忆提供“我是谁”，技能提供“我还能做什么”。

> 记忆 详细内容请参考见 [Memory](index/langchain-index/deepagent/concepts/memory.md)
> 技能 详细内容请参考见 [Skills](Skill.md)

## 配置文件：模型感知的行为包

`HarnessProfile` 允许你将系统提示调整、工具描述覆盖、工具和中件排除、额外中件、通用子代理设置等打包，并以提供商或模型为键进行注册。当创建代理时，框架会自动根据解析出的模型合并对应的配置文件。这使得“切换模型”不改变调用代码，同时让第三方可以以插件形式发布配置文件。

`ProviderProfile` 则专注于模型初始化参数（如 `temperature`）和凭据检查，仅在提供模型字符串时生效。两者配合，从模型构建到代理运行时行为均可集中管理。

> 配置文件 详细内容请参考见 [HarnessProfile](profile.md)

## 流式传输：可观测的代理树

Deep Agents 提供了两层流式传输 API。基础 **Streaming** 基于 LangGraph subgraph streaming，通过 `subgraphs=True` 启用后，每个事件都携带命名空间，标识来自主代理或哪个子代理。你可以同时订阅 `updates`、`messages`、`custom` 多种模式，构建丰富的实时监控。

更高级的**事件流式传输**（v0.6+）提供了类型化投影：`stream.subagents` 为每个子代理返回独立的流句柄，你可以在上面访问 `messages`、`tool_calls`、`output` 等字段，无需手动解析命名空间。这极大简化了多代理 UI 的构建。

> 流式传输 详细内容请参考见 [Streaming](index/langchain-index/deepagent/concepts/streaming.md)
> 事件流式传输 详细内容请参考见 [Event Streaming](stream_event.md)

## 全局关联的脉搏

纵观整个框架，虚拟文件系统是一切持久化和上下文流动的基础,文件权限配置负责虚拟文件系统的安全性问题；上下文工程将所有资源（文件、记忆、技能、工具）编织进代理的每一次思考；子代理将任务隔离与专业化变为现实，而流式传输让我们能够看见这一切的进行；配置文件则让整个系统在不同模型间优雅适配。当你理解了这些脉络，Deep Agents 就不再是一堆离散参数的集合，而是一个可以让代理持续、安全、智能工作的有机体。
