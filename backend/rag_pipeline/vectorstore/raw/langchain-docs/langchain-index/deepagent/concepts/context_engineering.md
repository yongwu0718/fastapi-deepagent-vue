# 上下文工程深度索引

> 这是 Deep Agents 上下文工程领域的**概念地图**，系统梳理了所有上下文类型、注入机制、压缩策略、隔离方式和长期持久化方案。  
> 阅读本文档可一次性掌握上下文相关概念之间的关联，并为后续深入细节提供导航。

---

## 概念全景

上下文工程的目标：**以正确的格式提供正确的信息和工具，让你的深度代理能够可靠地完成任务**。

深度代理可访问的上下文按作用域和生命周期分为五类：

| 上下文类型 | 控制内容 | 作用域 |
|-----------|---------|--------|
| **输入上下文** | 代理启动时注入其提示的内容（系统提示、记忆、技能） | 静态，每次运行都会应用 |
| **运行时上下文** | 调用时传入的静态配置（用户元数据、API 密钥、连接） | 每次运行，会传播到子代理 |
| **上下文压缩** | 内置的卸载和摘要功能，使上下文保持在窗口限制内 | 自动，当接近限制时触发 |
| **上下文隔离** | 使用子代理隔离繁重工作，仅将结果返回给主代理 | 每个子代理，在被委派时 |
| **长期记忆** | 使用虚拟文件系统实现跨线程持久存储 | 跨会话持久化 |

这些类型并不是孤立的，它们通过**运行时 (Runtime)**、**中间件 (Middleware)** 和**虚拟文件系统**相互关联，形成一个动态的上下文管理系统。

## 1. 输入上下文

代理启动时注入其系统提示的内容，决定了代理的初始知识和行为边界。

### 1.1 系统提示
- **组成**：自定义系统提示 + 内置代理指导（规划、文件系统、子代理等）
- **动态能力**：可通过 `@dynamic_prompt` 中间件从 `runtime.context` 或 `runtime.store` 读取用户偏好、角色等，动态生成提示。
- **关联配置**：使用 `HarnessProfile` 的 `base_system_prompt` / `system_prompt_suffix` 可为特定模型调整系统提示。

### 1.2 [记忆](index/langchain-index/deepagent/concepts/memory.md)
- **加载策略**：**始终全量加载**，不使用渐进式披露。
- **用途**：项目惯例、用户偏好、全局指南。
- **存储**：依赖虚拟文件系统，路径由后端决定。
- **与技能的区别**：记忆总是注入，技能按需加载。
- **最佳实践**：保持记忆精简，避免上下文过载；将详细工作流程放入技能。

### 1.3 [技能](Skill.md)
- **加载策略**：渐进式披露——启动时仅读取 `SKILL.md` 前置元数据，需要时才加载完整内容。
- **用途**：专业化工作流、领域知识。
- **关联**：技能内容可引用其他文件，这些文件存放在虚拟文件系统中。
- **设计原则**：每个技能专注单一工作流，避免宽泛重叠。

### 1.4 工具提示
- **来源**：
  - 内置工具（规划、文件系统、子代理等）通过中间件自动附加使用说明。
  - 用户自定义工具通过工具描述（docstring）提供指导。
- **定制**：可通过 `HarnessProfile.tool_description_overrides` 覆盖工具描述，或通过 `excluded_tools` 完全隐藏工具。
- **关联**：工具提示的质量直接影响模型对文件系统、子代理等能力的正确使用。

### 1.5 完整系统提示的组装顺序
最终的 `SystemMessage` 由以下部分组成，理解该顺序有助于调试上下文注入问题：

1. 自定义 `system_prompt`
2. 基础代理提示
3. 待办事项清单提示
4. 记忆提示（若提供）
5. 技能提示（若提供）
6. 虚拟文件系统提示（文件系统 + 执行工具文档）
7. 子代理提示
8. 用户自定义中间件提示
9. 人机协同提示（若设置 `interrupt_on`）

## 2. 运行时上下文

运行时上下文是每次调用时传入的静态配置，它不会自动注入模型提示，而是通过工具或中间件显式读取。

### 2.1 数据模型
- 使用 `context_schema` 定义（`dataclass` 或 `TypedDict`）。
- 通过 `agent.invoke` 的 `context` 参数传入。

### 2.2 访问方式
- **工具中**：通过注入的 `ToolRuntime` 对象，如 `runtime.context.user_id`。
- **中间件中**：通过 `request.runtime.context` 或直接作为参数注入。

### 2.3 作用域
- 运行时上下文**会传播到所有子代理**，确保子代理拥有相同的配置。
- 子代理内部还可通过命名空间键访问其专有上下文。

### 2.4 典型用途
用户 ID、API 密钥、功能标志、权限角色等。配合动态系统提示，可实现基于角色的工具和指令切换。

## 3. 上下文压缩

长时间运行任务产生的海量工具输出和消息历史可能导致上下文窗口溢出。压缩机制通过两种内置手段保持上下文在限制内：

### 3.1 卸载
- **触发条件**：工具调用输入或结果超过令牌阈值（默认 20,000 令牌）。
- **行为**：
  - 输入过大 → 旧工具调用被截断，替换为文件系统引用。
  - 结果过大 → 完整结果写入文件系统，对话中仅保留文件路径和前 10 行预览。
- **依赖**：虚拟文件系统（代理可随时 `read_file` 或 `grep` 检索已卸载内容）。

### 3.2 摘要
- **触发条件**：当上下文大小超过模型 `max_input_tokens` 的 85%，且无更多内容可卸载时。
- **双重机制**：
  - **上下文内摘要**：LLM 生成结构化摘要（意图、工件、后续步骤）替换完整历史。
  - **文件系统保存**：完整消息历史作为规范记录写入文件系统。
- **配置**：保留 10% 令牌作为近期上下文；回退阈值 170,000 令牌 / 6 条消息；遇到 `ContextOverflowError` 立即重试摘要。
- **可选摘要工具**：可添加 `SummarizationToolMiddleware` 让代理在任务间隙主动触发摘要，而不完全依赖令牌阈值。

## 4. 上下文隔离 [子代理](index/langchain-index/deepagent/concepts/subagent.md)

子代理是解决上下文膨胀的核心模式，通过将繁重任务隔离到独立上下文，主代理仅接收最终报告。

### 4.1 工作原理
- 主代理通过 `task` 工具委派任务。
- 子代理拥有全新的上下文（独立的系统提示、工具、技能等），执行完毕后返回单一结果。
- 主代理的上下文保持干净，不会积累中间步骤。

### 4.2 最佳实践
- **委派复杂任务**：多步骤、输出量大的工作交给子代理。
- **强制摘要输出**：在子代理的 `system_prompt` 中明确要求只返回摘要（<500 字），避免长报告。
- **使用文件系统传递大数据**：子代理可将结果写入文件，主代理按需读取。

### 4.3 与其他模块的关系
- **上下文压缩**：子代理隔离是上下文压缩的高级形式，避免了主代理的卸载/摘要开销。
- **运行时上下文**：子代理继承父代理的运行时配置，确保一致性。
- **人机协同**：可对子代理的特定工具设置中断。

## 5. 长期记忆

长期记忆将数据持久化到跨会话的存储中，使代理能够跨对话保留信息。

### 5.1 实现方式
- 使用 `CompositeBackend` 将特定路径（如 `/memories/`）路由到 `StoreBackend`（LangGraph Store）。
- 其余文件仍使用 `StateBackend`（会话内）。

### 5.2 使用模式
- 在系统提示中明确告知代理应保存什么、保存在哪里（例如：“当用户提到偏好时，保存到 `/memories/preferences.txt`”）。
- 代理使用标准文件系统工具（`write_file`、`edit_file`）操作长期记忆路径。
- 无需预先填充文件，代理按需创建。

### 5.3 与记忆和技能的区别
- **记忆 (Memory)**：属于输入上下文，每次全量加载，通常位于项目级 `AGENTS.md`，内容相对固定。
- **长期记忆**：是代理在运行过程中主动写入的持久化数据，存储用户特定信息或经验，可动态增长。
- **技能**：提供流程化能力，不用于存储用户数据。

## 全局关联总结

上下文工程并非孤立模块，它贯穿整个 Deep Agents 框架：

- **[虚拟文件系统](index/langchain-index/deepagent/concepts/backends.md)** 是所有上下文存储的基石（记忆、技能、卸载、摘要存档、长期记忆）。
- **[子代理](index/langchain-index/deepagent/concepts/subagent.md)** 提供了上下文隔离的实际机制，与上下文压缩相辅相成。
- **[记忆](index/langchain-index/deepagent/concepts/memory.md)** 和 **[技能](Skill.md)** 是输入上下文的两大支柱，加载策略相反。
- **运行时 Runtime** 是贯穿输入、工具、压缩和记忆的统一注入总线。
- **中间件** 是实现动态上下文（动态提示、动态工具、动态模型选择、摘要等）的执行引擎。
- **框架配置文件 (HarnessProfile)** 允许按模型/提供商调整系统提示和工具集，影响全局上下文行为。

---

## 最佳实践速查

1. **输入上下文**：记忆精简，技能专注。
2. **运行时上下文**：用于用户元数据和密钥，传播到子代理。
3. **压缩**：依赖内置卸载和摘要，必要时加入主动摘要工具。
4. **隔离**：为繁重任务创建子代理，并约束其输出长度。
5. **长期记忆**：明确定义存储结构和 agent 的读写策略。
6. **文件系统**：让代理善用文件读写来替代长对话，减少上下文占用。
