# 记忆深度索引

> 这是 Deep Agents 记忆系统的**概念地图**，涵盖记忆工作原理、作用域（代理/用户/组织）、更新策略、后台合并、只读与可写以及多代理隔离。

## 概念全景

记忆让代理能够跨对话学习和改进。Deep Agents 将记忆视为文件系统中的文件，通过后端控制存储位置和访问隔离。核心机制：

1. **指向记忆文件**：创建代理时通过 `memory=` 指定文件路径。
2. **代理读取记忆**：启动时加载到系统提示，或按需读取（技能采用渐进式披露）。
3. **代理更新记忆**（可选）：通过 `edit_file` 工具写入，可发生在对话期间或后台合并。

### 关键区分：记忆 vs 技能 vs 长期记忆

| 概念 | 定位 | 加载策略 | 典型用途 |
|------|------|---------|---------|
| **记忆 (Memory)** | 持久上下文（偏好、惯例） | **始终全量加载** | 用户偏好、项目指南 |
| **技能 (Skills)** | 程序性指令（工作流） | **渐进式披露**（启动时仅读元数据，需要时加载全文） | 专门化任务能力 |
| **长期记忆 (Long-term Memory)** | 代理在运行中主动写入的持久化数据 | 存储在后端，跨会话可用 | 用户特定信息、经验积累 |

## 1. 作用域模型

记忆的作用域通过后端命名空间控制，决定谁可以读取和写入。

### 代理作用域记忆
- 所有用户共享同一份记忆文件，代理随时间演变其“身份”和累积知识。
- 命名空间使用 `assistant_id`，确保同一代理的所有对话访问相同文件。
- 适合：公共助手、专职代理角色。

### 用户作用域记忆
- 每个用户拥有独立的记忆文件，代理按用户隔离偏好和历史。
- 命名空间使用 `user_id`，杜绝跨用户泄漏。
- 适合：多用户应用，个性化体验。

### 组织作用域记忆
- 应用于组织内所有用户的策略或共享知识。
- 通常**只读**，通过应用代码或 Store API 填充，防止提示注入。
- 命名空间使用 `org_id` 等组织标识符，与用户作用域记忆可共存。

**命名空间组合**：可按 `(assistant_id, user_id)` 实现每代理每用户隔离，或单独使用 `assistant_id` 进行代理级共享。

## 2. 后端与路由

记忆的物理存储由后端决定，通常使用 `CompositeBackend` 将记忆路径路由到持久化存储：

```python
CompositeBackend(
    default=StateBackend(),          # 工作文件：线程作用域
    routes={
        "/memories/": StoreBackend(namespace=...),   # 记忆：跨会话持久
        "/skills/": StoreBackend(namespace=...),     # 技能：跨会话持久
        "/policies/": StoreBackend(namespace=...),   # 组织策略：跨会话持久
    }
)
```

- `StateBackend`：线程内短期存储，不适用于记忆。
- `StoreBackend`：LangGraph Store 提供跨会话持久化，是记忆的标准后端。
- `ContextHubBackend`：可替代 StoreBackend，将记忆持久化到 LangSmith Hub 仓库。

## 3. 记忆更新策略

### 热路径（默认）
- 代理在对话期间直接通过 `edit_file` 工具写入记忆。
- 优点：即时生效，用户透明。
- 缺点：增加延迟，代理需多任务处理。

### 后台合并
- 在对话之间由独立的**合并代理**处理记忆更新。
- 合并代理读取最近的对话历史，提取关键事实，合并到记忆存储。
- 通过 cron 定时触发，使用 LangGraph SDK 的 `client.crons.create`。
- 适合：减少延迟、综合多对话信息、避免热路径冲突。

**合并代理设计要点**：
- 使用 `search_recent_conversations` 工具获取指定时间窗口内的对话。
- 系统提示指导提取、合并、删除过时信息。
- cron 间隔须与工具的回溯窗口匹配，避免漏处理或重复处理。

---

## 4. 只读与可写记忆

| 模式 | 实现方式 | 适用场景 |
|------|---------|---------|
| **可读写**（默认） | 代理通过 `edit_file` 更新 | 用户偏好、自我改进 |
| **只读** | 使用 `permissions` 拒绝写入特定路径，或通过策略钩子控制 | 组织策略、合规规则、共享知识库 |

**安全实践**：
- 默认使用用户作用域以避免跨用户注入。
- 共享策略设只读，仅通过应用代码填充。
- 对写入共享记忆的操作添加人机协同（`interrupt_on`）。

---

## 5. 并发与多代理隔离

- **并发写入**：用户作用域下很少并发冲突；代理作用域或组织作用域建议用后台合并序列化写入，或将记忆拆分为多个文件减少争用。
- **同一部署多个代理**：在命名空间中包含 `assistant_id`，实现每个代理独立的记忆空间。

---

## 6. 情景记忆

情景记忆存储对话过程和顺序，不同于语义记忆（事实/偏好）。Deep Agents 利用检查点器持久化对话线程。通过包装 `client.threads.search` 为工具，代理可搜索过去的对话历史，获取相关上下文。可按用户或组织过滤元数据。

## 与全局概念的关联

- **[虚拟文件系统与后端](index/langchain-index/deepagent/concepts/backends.md)**：记忆是文件系统之上的应用，依赖 `CompositeBackend` 路由和 `StoreBackend` 实现持久化；权限可控制写入。
- **[技能](Skill.md)**：与记忆共同构成输入上下文，但加载策略相反（记忆全量、技能按需）；两者可存储在同一 `StoreBackend` 的不同路径。
- **[上下文工程](index/langchain-index/deepagent/concepts/context_engineering.md)**：记忆属于输入上下文，通过 `@dynamic_prompt` 可动态注入；长期记忆是上下文工程中 Tool Context 写入的典型产物。
- **[权限](index/langchain-index/deepagent/concepts/permissions.md)**：可用于强制只读记忆，阻止代理修改特定路径。
- **[人机协同](index/langchain-index/deepagent/concepts/Human-in-the-loop.md)**：在写入共享记忆时可结合中断进行人工审批，防止恶意注入。
- **[子代理](index/langchain-index/deepagent/concepts/subagent.md)**：子代理继承主代理的记忆和技能配置，但可通过 `skills` 参数赋予独立技能；通用子代理自动继承记忆和技能。
