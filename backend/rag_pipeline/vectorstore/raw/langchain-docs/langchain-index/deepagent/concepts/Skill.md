# 技能深度索引

> 这是 Deep Agents 技能系统的**概念地图**，涵盖技能定义、渐进式披露、配置与后端集成、解释器技能、沙箱脚本执行、子代理继承以及技能与记忆/工具的边界。  
> 阅读本文档可一次性掌握技能领域的全部概念及其在代理能力扩展中的角色。

---

## 概念全景

技能是**可复用的代理能力**，提供专业化的工作流程和领域知识。每个技能是一个包含 `SKILL.md` 文件的目录，可附带脚本、模板、参考文档等资源。技能遵循 Agent Skills 规范，并通过**渐进式披露**按需加载——启动时只读取前置元数据中的描述，只有当代理判断技能相关时才读取完整内容，从而节省上下文。

### 技能 vs 记忆 vs 工具

| 维度 | 技能 (Skills) | 记忆 (Memory) | 工具 (Tools) |
|------|--------------|--------------|-------------|
| **目的** | 程序性指令与领域知识 | 持久偏好与惯例 | 与外部系统交互 |
| **加载策略** | 渐进式披露（按需） | 始终全量注入系统提示 | 始终可用（通过工具描述） |
| **格式** | `SKILL.md` + 可选资源 | `AGENTS.md` 文件 | Python/JS 函数 |
| **适用场景** | 大型、任务特定指令；捆绑能力和上下文 | 跨对话恒定偏好和指南 | 单一、明确的操作（API 调用、数据库查询等） |

## 1. 技能构成与规范

每个技能至少包含一个 `SKILL.md`，其结构为：

- **前置元数据 (YAML frontmatter)**：`name`、`description`（必填；<1024 字符，超过截断）、可选 `license`、`compatibility`、`metadata`、`allowed-tools`、`module` 等。
- **正文**：Markdown 格式的指令，告诉代理何时及如何使用该技能，以及如何访问附带资源。

**关键约束**：
- 在 Deep Agents 中，`SKILL.md` 文件必须小于 **10 MB**，超出则被跳过。
- `description` 是代理进行技能匹配的唯一依据，务必清晰、具体。

**完整示例**：

```markdown
---
name: langgraph-docs
description: 对于与 LangGraph 相关的请求，使用此技能获取相关文档以提供准确指导。
license: MIT
compatibility: 需要互联网连接
metadata:
  author: langchain
  version: "1.0"
allowed-tools: fetch_url
module: index.ts
---

# langgraph-docs

## 概述
...
## 指令
### 1. 获取文档索引
使用 fetch_url 工具读取 https://docs.langchain.com/llms.txt
...
```

## 2. 渐进式披露机制

1. **启动时**：代理读取每个 `SKILL.md` 的前置元数据（名称、描述等），但不加载正文。
2. **收到提示时**：代理根据描述判断哪些技能可能相关。
3. **匹配后**：代理从文件系统读取完整的 `SKILL.md` 正文及引用的资源。
4. **执行**：遵循技能指令，按需使用附带脚本、模板或解释器模块。

配置技能后，系统提示中会注入“技能系统”部分，引导代理执行上述三步流程。

## 3. 配置与后端集成

### 基本用法
创建代理时通过 `skills` 参数指定技能源路径（以正斜杠分隔，相对后端根目录）：

```python
agent = create_deep_agent(
    model="...",
    backend=backend,
    skills=["/skills/"],
    # 如果使用解释器技能，需添加 CodeInterpreterMiddleware
    middleware=[CodeInterpreterMiddleware(skills_backend=backend)],
)
```

### 后端选择与技能存储
- **StateBackend（默认）**：通过 `invoke(files={...})` 传入技能文件，使用 `create_file_data()` 格式化。
- **StoreBackend**：技能文件先存入 LangGraph Store，再通过命名空间访问。适合跨会话持久化。
- **FilesystemBackend**：直接从本地磁盘加载技能，路径相对于 `root_dir`。
- **CompositeBackend**：可将技能路径路由到特定后端（如 `/skills/` → `StoreBackend`），与工作目录分离。

### 源优先级
当多个技能源包含同名技能时，列表中**靠后的源优先**（后者胜出），允许分层覆盖（用户技能覆盖项目技能等）。

## 4. 解释器技能（可导入代码模块）

**解释器技能**向 QuickJS 解释器暴露可导入的代码模块，使代理能直接调用经过测试的助手函数，避免让模型重复生成代码。

### 启用方式
1. 在 `SKILL.md` 前置元数据中添加 `module` 键，指向技能目录下的 JS/TS 文件。
2. 创建代理时添加 `CodeInterpreterMiddleware(skills_backend=backend)`。
3. 代理在解释器代码中通过 `await import("@/skills/<技能名>")` 导入模块。

**适用场景**：可复用的、确定性的逻辑，如解析器、规范化器、聚合例程、验证器等，适合作为工作流中的构建块。

---

## 5. 在沙箱中执行技能脚本

当技能包含需要 Shell 执行的脚本（如 Python 文件）时，必须使用**沙箱后端**。

### 复合后端下的文件同步问题
若技能存储在 `StoreBackend`（持久化），而执行环境是沙箱（默认后端），则技能文件在沙箱中不可见。需通过**自定义中间件**在每次代理运行前将技能文件从存储上传到沙箱。

**模式**：
- 实现 `AgentMiddleware`，在 `abefore_agent` 中读取共享存储命名空间中的技能文件，调用 `backend.aupload_files()` 批量上传。
- 部署时通过种子脚本将技能文件存入共享命名空间。

## 6. 子代理与技能继承

- **通用子代理**（`general-purpose`）：自动继承主代理的所有技能，无需额外配置。
- **自定义子代理**：**不**继承主代理技能。需在子代理定义中通过 `skills` 字段显式赋予技能集。技能状态完全隔离——主代理的技能对子代理不可见，反之亦然。

```python
subagents=[{
    "name": "researcher",
    "skills": ["/skills/research/", "/skills/web-search/"],  # 独立技能集
}]
```

## 7. 最佳实践

- **编写精准的描述**：`description` 是代理匹配技能的唯一依据，要具体说明何时使用。
- **单个技能专注单一工作流**：避免宽泛重叠，防止误匹配和上下文膨胀。
- **将详细参考资料外置**：在 `SKILL.md` 中引用外部脚本、文档，保持主指令精炼。
- **利用解释器技能封装确定性逻辑**：减少模型生成代码的不确定性，提升可靠性。
- **为沙箱执行使用专用同步中间件**：确保持久化技能在沙箱可用。
- **使用 LangSmith 追踪技能匹配与执行**：观察代理如何发现和运用技能。

## 与全局概念的关联

- **[虚拟文件系统与后端](index/langchain-index/deepagent/concepts/backends.md)**：技能存储于虚拟文件系统，依赖后端实现加载和持久化；`CompositeBackend` 可隔离技能存储路由。
- **[上下文工程](index/langchain-index/deepagent/concepts/context_engineering.md)**：技能的渐进式披露是动态上下文管理的典型案例，通过按需加载控制令牌消耗。
- **[记忆](index/langchain-index/deepagent/concepts/memory.md)**：技能与记忆同为输入上下文来源，但加载策略相反（技能按需，记忆全量）；可在同一后端不同路径下共存。
- **[子代理](index/langchain-index/deepagent/concepts/subagent.md)**：通用子代理继承技能，自定义子代理需独立配置，实现委派任务的专业化。
- **[沙箱](sandboxs.md)和[解释器](index/langchain-index/deepagent/concepts/Interpreters.md)**：解释器技能依赖解释器中间件；沙箱脚本执行依赖沙箱后端和同步中间件。
- **[人机协同与权限](index/langchain-index/deepagent/concepts/Human-in-the-loop.md)**：对敏感技能的操作，可结合中断机制与权限规则进行控制。

## 链接原文

当本索引中的概要无法满足你（例如需要完整代码实现、方法签名、罕见配置示例）时，请通过以下方式从原始文档中获取精确信息。

### 语义检索（聚焦查询）

原始文档已按 `#` 级别标题切分并向量化。构造查询时，**使用当前索引章节的标题或段落内出现的关键概念、特殊术语作为锚点**，而不是全文反复出现的通用词。有效的查询往往短而具体。

例如，当你在本索引的“解释器技能”一节需要更多细节时：

- **好的查询**：`解释器技能 module 键 可导入代码模块`、`await import 解释器技能 CodeInterpreterMiddleware`、`确定性逻辑 解析器 规范化器`
- **差的查询**：`怎么使用技能`（整个文档都在讲技能，无法聚焦）

将标题词和段落内的特有术语组合，可以快速锁定目标段落。

### 利用索引页提升检索精度

如果单靠关键术语检索结果仍不够集中，从本索引中提取**所在章节的标题**或**当前段落的特有表述**作为附加上下文，与你的问题组合成更完整的查询。索引页的标题本身就是高质量的语义锚点。例如：

- 想了解“渐进式披露”的三步流程具体实现，用 `渐进式披露 启动时 前置元数据 匹配后` 组合查询。
- 想了解“沙箱中执行技能脚本”的同步中间件模式，用 `沙箱脚本执行 复合后端 文件同步 自定义中间件 abefore_agent` 定位。
- 想查询“技能源优先级”中“后者胜出”的具体覆盖逻辑，用 `源优先级 后者胜出 分层覆盖` 找到说明。

### 标题路径兜底

语义检索返回的每个片段都携带其**原文标题和文件路径**。若需读取该章节的完整内容或进入相邻段落，可直接用返回结果中的标题坐标通过 `read_file` 精确定位——标题始终精确，因为它来自原文本身。
