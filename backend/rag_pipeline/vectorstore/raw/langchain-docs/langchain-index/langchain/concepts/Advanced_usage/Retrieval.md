# Retrieval

> 这是 LangChain 中**检索（Retrieval）与检索增强生成（RAG）**的胖索引，覆盖知识库构建、检索管道、RAG 架构（2步、Agentic、混合型）及其实践决策。
> 阅读本文档可一次性掌握检索领域的全部概念及其关联，为构建具备外部知识访问能力的 Agent 提供架构支撑。

---

## 概念全景

检索通过在运行时获取相关外部知识，弥补 LLM 上下文窗口有限和知识静态的短板。它是检索增强生成（RAG）的基础，使模型能够基于最新、特定领域的信息生成有依据的回答。

| 环节             | 描述                                                         | 关键组件 / 架构                      |
| ---------------- | ------------------------------------------------------------ | ------------------------------------ |
| **知识准备**     | 将文档转化为可搜索的向量存储                                 | Document loaders, text splitters, embeddings, vector stores |
| **检索过程**     | 根据查询获取相关文档片段                                     | Retriever 接口, 查询嵌入, 相似度搜索 |
| **生成集成**     | 将检索结果注入 LLM 的上下文中以生成答案                      | 2步 RAG, Agentic RAG, 混合型 RAG     |
| **高级控制**     | 查询增强、检索验证、答案验证，通过迭代提升质量               | 混合型架构, 多轮检索                 |

核心决策点：**选择哪种 RAG 架构（简单可预测的 2 步 RAG，还是灵活的 Agentic RAG，或带验证的混合型）**，以及**如何构建或接入知识库**，共同决定了系统的可靠性、延迟和可扩展性。

## 1. 知识库与检索管道

### 构建知识库

通过文档加载器、文本分割器、嵌入模型和向量存储，将私有数据（PDF、网页、Notion 等）转化为可搜索的知识库。

- **文档加载器**：从不同数据源摄取数据，输出标准化的 `Document` 对象。
- **文本分割**：将长文档拆分为适合上下文窗口的 chunk。
- **嵌入模型**：将文本转换为向量，使语义相近的内容在向量空间中靠近。
- **向量存储**：专用数据库，用于存储和快速检索嵌入向量。

### 使用已有知识库

若已有 SQL 数据库、CRM 或内部文档系统，无需重建，可直接：
- 作为 **工具** 接入 Agent，在 Agentic RAG 中动态查询。
- 通过简单的 2 步流程：先检索，再作为上下文传给 LLM。

### 检索器接口

`Retriever` 是统一抽象，根据非结构化查询返回相关文档列表。检索管道可以灵活组合各种检索器（向量检索、关键词检索、混合检索）。

## 2. RAG 架构对比

| 架构         | 流程特点                                                     | 控制性 | 灵活性 | 延迟       | 适用场景                               |
| ------------ | ------------------------------------------------------------ | ------ | ------ | ---------- | -------------------------------------- |
| **2步 RAG**  | 固定顺序：检索 → 生成；LLM 调用次数已知且有上限              | 高     | 低     | 可预测、快 | FAQ、文档问答机器人                   |
| **Agentic RAG** | Agent 自主决定何时检索、用何工具、如何迭代，直到能生成答案   | 低     | 高     | 可变       | 研究助手、多工具复杂任务               |
| **混合型 RAG** | 结合两者：加入查询增强、检索验证、答案验证等中间步骤，支持迭代优化 | 中等   | 中等   | 可变       | 需质量控制的特定领域 Q&A、合规场景     |

### 2步 RAG

最简单的 RAG 模式。每次用户查询，系统先检索相关文档，再将文档与问题拼接传给 LLM 生成答案。流程固定，延迟可预测。适合对可靠性和响应速度要求高的场景。

### Agentic RAG

Agent 可访问一个或多个检索工具（如搜索引擎、文档抓取器、数据库查询）。它通过推理循环逐步获取信息，可多次检索、交叉验证，最终生成带引用的答案。

关键要素：Agent 必须能调用获取外部知识的**工具**。示例中 `fetch_url` 或 `fetch_documentation` 作为工具，让模型在需要时主动检索。

### 混合型 RAG

在 2 步 RAG 基础上增加控制循环：
- **查询增强**：改写模糊查询、生成多角度变体。
- **检索验证**：判断检索结果是否足够相关、充分，若不充分则优化查询重新检索。
- **答案验证**：检查生成答案的准确性、完整性、是否与源文档一致；可能触发重新生成或人工介入。

这种架构在保持一定可控性的同时，通过迭代提升输出质量。

## 3. 最佳实践

- **模块化设计**：加载器、分割器、嵌入、存储、检索器均可独立替换，不影响应用逻辑。
- **从简开始**：先用 2 步 RAG 验证效果，再根据需要引入 Agentic 或混合型。
- **延迟意识**：2 步 RAG 的延迟可预测；Agentic RAG 因多次 LLM 调用和工具执行，延迟可能较高，需权衡。
- **检索质量直接影响生成质量**：投入精力优化 chunk 大小、嵌入模型选择和检索策略。
- **工具安全**：在 Agentic RAG 中限制可访问的域名或数据库表，防止越权。
- **引用溯源**：让 LLM 引用检索到的具体片段，便于用户验证。
- **评估与迭代**：定期评估检索召回率和答案准确率，调整检索参数或架构。

---

## 4. 与全局概念的关联

- **工具 (Tools)**：在 Agentic RAG 中，检索器、文档抓取器、数据库查询等均以工具形式暴露给 Agent，供模型主动调用。
- **模型 (Models)**：检索到的内容作为上下文注入模型调用，上下文工程通过系统提示或消息修改来实现；模型选择影响最大上下文窗口和推理能力。
- **消息 (Messages)**：检索结果通常作为 `HumanMessage` 或 `SystemMessage` 追加到对话历史中。
- **短期记忆 (Short-term memory)**：对话过程中缓存的检索结果可存入 State，避免重复检索。
- **长期记忆 (Store)**：常用知识点或用户偏好可持久化，减少外部检索延迟。
- **上下文工程**：检索是动态上下文的典型来源；通过中间件在模型调用前注入检索结果，与动态提示、消息注入等模式一致。
- **护栏 (Guardrails)**：对检索结果进行过滤（如 PII 脱敏、有害内容屏蔽），确保注入上下文的安全性。
