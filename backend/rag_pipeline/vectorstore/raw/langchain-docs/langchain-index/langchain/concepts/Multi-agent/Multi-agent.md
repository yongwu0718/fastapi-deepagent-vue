# Multi-agent

> 这是 LangGraph 中**多智能体系统**的胖索引，覆盖五大协作模式（Subagents、Handoffs、Skills、Router、Custom workflow）的工作原理、选择矩阵、性能特征与上下文工程核心。
> 阅读本文档可一次性掌握多智能体设计的全部概念及其关联，为架构高内聚、低耦合的复合 Agent 系统提供决策支撑。

---

## 概念全景

多智能体系统通过协调多个专门的 Agent 来处理复杂工作流。其核心设计问题是**上下文工程**：每个 Agent 应该看到什么信息才能高效完成其任务，同时避免上下文过载。

| 需求动因               | 说明                                                         |
| ---------------------- | ------------------------------------------------------------ |
| **上下文管理**         | 将专业知识分布在多个 Agent 中，避免单个提示窗口过载         |
| **分布式开发**         | 不同团队独立开发维护各自的 Agent，通过清晰的边界组合         |
| **并行化**             | 为子任务生成专门的 Worker 并并发执行，加速整体流程           |

五种主要协作模式：

| 模式                    | 工作机制                                                     | 控制流               | 状态特征           |
| ----------------------- | ------------------------------------------------------------ | -------------------- | ------------------ |
| **Subagents**           | 主 Agent 将子代理作为工具进行协调，所有路由经过主 Agent     | 中央调度             | 子代理默认无状态   |
| **Handoffs**            | Agent 通过工具调用直接将控制权和状态移交给其他 Agent         | Agent 间直接传递     | 可跨轮次保持状态   |
| **Skills**              | 按需加载专业提示和知识，一个 Agent 保持主导控制权             | 单 Agent + 动态注入  | 技能上下文一旦加载即持续 |
| **Router**              | 路由步骤对输入分类，导向一个或多个专门的 Agent，结果综合返回 | 分类 → 分发 → 聚合   | 路由本身无状态     |
| **Custom workflow**     | 使用 LangGraph 构建混合确定性逻辑与 Agent 行为的特定执行流程 | 图定义的控制流       | 全图状态管理       |

核心决策点：**根据任务是否可并行、是否需要跨轮记忆、是否涉及多领域大上下文、团队是否需独立开发，选择一种或多种模式的组合**。

## 1. 模式详解与适用性矩阵

| 能力维度             | Subagents | Handoffs | Skills | Router |
| -------------------- | :-------: | :------: | :----: | :----: |
| **分布式开发**       | ⭐⭐⭐⭐⭐   |    -     | ⭐⭐⭐⭐⭐ |  ⭐⭐⭐  |
| **并行化**           | ⭐⭐⭐⭐⭐   |    -     |  ⭐⭐⭐  | ⭐⭐⭐⭐⭐ |
| **多跳（串联调用）** | ⭐⭐⭐⭐⭐   | ⭐⭐⭐⭐⭐  | ⭐⭐⭐⭐⭐ |   -    |
| **直接用户交互**     |     ⭐     | ⭐⭐⭐⭐⭐  | ⭐⭐⭐⭐⭐ |  ⭐⭐⭐  |

- **Subagents**：通过工具调用的方式实现最完整的中央控制与隔离，并行能力强，适合需要严格上下文隔离和团队独立开发的场景。
- **Handoffs**：Agent 间像交接棒一样传递控制权，天然支持多轮对话状态保持，适合面向用户的交互式多步工作流。
- **Skills**：最轻量的专业化方式，不增加 Agent 数量，仅动态注入提示和知识，适合简单聚焦的任务。
- **Router**：适合需要明确分类和分发的场景，通过 LLM 进行路由决策，可并行调用多个后端 Agent。

可以混合使用这些模式，例如在 Subagents 架构中调用 Router Agent 作为工具，或让 Subagent 使用 Skills 加载专业上下文。

## 2. 性能特征

### 单次请求

| 模式        | 模型调用次数 | 延迟特征               |
| ----------- | :----------: | ---------------------- |
| Subagents   |      4       | 多一层中央调度         |
| Handoffs    |      3       | 直接交接，最简链路     |
| Skills      |      3       | 无额外调度调用         |
| Router      |      3       | 路由 LLM 调用 + Agent 调用 |

### 重复请求（同一对话中再次执行相同任务）

| 模式        | 第2轮调用次数 | 总计     | 原因                         |
| ----------- | :----------: | :------: | ---------------------------- |
| Subagents   |      4       |    8     | 子代理每次调用都是全新无状态 |
| Handoffs    |      2       |    5     | Agent 保持活跃，跳过交接     |
| Skills      |      2       |    5     | 技能上下文已加载，无需重复   |
| Router      |      3       |    6     | 路由本身无状态，需重新分类   |

**有状态模式（Handoffs、Skills）在重复请求中可节省 40-50% 调用次数。**

### 多领域任务（并行比较多个专业领域）

| 模式        | 模型调用次数 | 总 Tokens  | 关键特征                         |
| ----------- | :----------: | :-------: | -------------------------------- |
| Subagents   |      5       |   ~9K     | 隔离环境，仅处理各自上下文       |
| Handoffs    |     7+       |  ~14K+    | 顺序执行，对话历史累积开销大     |
| Skills      |      3       |   ~15K    | 调用次数少，但全部技能文档占用上下文 |
| Router      |      5       |   ~9K     | 路由后并行调用，与 Subagents 类似 |

**对于多领域并行任务，Subagents 和 Router 的 Token 效率最高（上下文隔离）；Skills 虽然调用次数少但单次上下文负载大。**

## 3. 关键约束与最佳实践

- **上下文工程是核心**：多智能体的质量取决于每个 Agent 是否恰好拥有完成任务所需的信息，不多不少。
- **优先考虑简单方案**：许多“需要多智能体”的问题通过单个 Agent 配合动态工具和提示也能解决；仅在明确需要隔离、并行或独立开发时才引入多智能体。
- **混合模式**：不存在唯一最优模式；根据具体工作流混合使用 Subagents、Handoffs、Skills 和 Router。
- **子代理的封装与测试**：通过 LangGraph 的 `checkpointer=True` 控制子代理是有状态还是无状态；无状态子代理易于并行和复用，有状态子代理适合需要跨轮记忆的场景。
- **性能调优**：根据任务特征（是否可并行、是否需要长对话记忆）选择模式以最小化 LLM 调用次数和 Token 消耗。
- **使用 LangSmith 追踪**：完整记录跨 Agent 的协调流程，便于调试和优化。

## 4. 与全局概念的关联

- **子图 (Subgraphs)**：Subagents 模式直接对应子图机制，每个子代理是一个 LangGraph 图；Handoffs 也可通过子图 + `Command(graph=PARENT)` 实现控制权转移。
- **工具 (Tools)**：Subagents 和 Skills 的核心机制是将子代理或技能封装为工具，由上层 Agent 通过 `bind_tools` 调用。
- **上下文工程 (Context Engineering)**：多智能体设计本质上是上下文工程——每个 Agent 的系统提示、消息、工具和模型都是动态决定的。
- **记忆 (Memory)**：Handoffs 依赖跨轮状态保持，需使用每线程 (Per-thread) 子图持久化；无状态 Subagents 则使用每次调用 (Per-invocation) 模式。
- **流式传输 (Streaming)**：通过 `subgraphs=True` 可流式输出子代理内部事件，实现多 Agent 交互的可观察性。
- **模型 (Models)**：不同子代理可使用不同模型，根据任务需求平衡成本与能力。
- **中断 (Interrupts)**：Handoffs 和 Subagents 均支持在子代理内部使用 `interrupt()`，实现精细的人机协同。
