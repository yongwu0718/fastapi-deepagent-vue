# Subagents

> 这是 LangGraph 多智能体系统中 **Subagents（子代理）** 模式的胖索引，覆盖中央调度、同步/异步执行、工具封装模式、上下文工程与状态管理。
> 阅读本文档可一次性掌握子代理设计的全部概念及其关联，为构建具有强上下文隔离和集中控制的多智能体应用提供决策支撑。

---

## 概念全景

子代理模式由一个**监督 Agent（主 Agent）** 通过工具调用协调多个无状态的子代理。主 Agent 维护完整的对话历史和上下文，子代理每次调用在干净的上下文窗口中工作，仅返回必要的结果，从而避免上下文膨胀。

| 维度               | 描述                                                         |
| ------------------ | ------------------------------------------------------------ |
| **控制流**         | 所有路由均经过主 Agent，由其决定调用哪个子代理、传递什么输入以及如何合并结果 |
| **子代理特征**     | 无状态（不记忆过去的交互），不直接与用户对话，结果返回给主 Agent |
| **封装方式**       | 每个子代理封装为一个工具，主 Agent 通过工具调用子代理         |
| **执行模式**       | 同步（主 Agent 等待结果）或异步（后台执行，主 Agent 继续交互） |
| **工具模式**       | “每个 Agent 一个工具”提供细粒度控制；“单个分发工具”实现约定优于配置 |
| **上下文工程**     | 精确控制子代理规格（名称/描述）、输入（传递何种上下文）和输出（返回什么信息） |

核心决策点：**同步还是异步、每个子代理独立工具还是统一分发工具、如何向子代理传递上下文、如何设计子代理的提示以得到期望的输出格式**。

## 1. 基本实现

核心是将子代理封装为工具：

```python
@tool("research", description="Research a topic and return findings")
def call_research_agent(query: str):
    result = subagent.invoke({"messages": [{"role": "user", "content": query}]})
    return result["messages"][-1].content

main_agent = create_agent(model="...", tools=[call_research_agent])
```

## 2. 同步 vs 异步

| 模式     | 主 Agent 行为                                 | 适用场景                         | 权衡                             |
| -------- | --------------------------------------------- | -------------------------------- | -------------------------------- |
| **同步** | 阻塞等待子代理完成后再继续                    | 结果强依赖子代理输出             | 简单，但长时间任务会冻结对话     |
| **异步** | 启动后台作业后立即返回，后续通过轮询获取结果   | 独立后台任务，用户无需等待       | 响应迅速，但需额外的作业管理逻辑 |

异步模式通常需要实现三个工具：启动作业、检查状态、获取结果。

## 3. 工具模式

| 模式                     | 描述                                                         | 适用场景                                 |
| ------------------------ | ------------------------------------------------------------ | ---------------------------------------- |
| **每个 Agent 一个工具**  | 为每个子代理创建独立工具，名称和描述由开发者精细控制         | 需要强定制、子代理数量少且固定           |
| **单个分发工具**         | 使用一个通用的 `task` 工具，通过参数指定调用哪个子代理       | 子代理数量多、团队独立开发、注册表动态变化 |

单个分发工具可进一步通过系统提示枚举、枚举约束或基于工具发现（提供 `list_agents` 工具）的方式让主 Agent 了解可用的子代理。

## 4. 上下文工程

### 子代理规格

- **名称与描述**：是主 Agent 决定调用的依据，必须清晰、以行动为导向。
- **发现机制**：静态列表（系统提示或枚举约束）适合小型固定集合；动态发现（`list_agents` 工具）适合大型或动态注册表。

### 子代理输入

可通过 `ToolRuntime` 访问主 Agent 的 state，有选择地传递消息历史、用户上下文或其他状态键，为子代理提供完成任务所需的最小上下文。

### 子代理输出

- 提示子代理在最终消息中包含所有必要信息（因为主 Agent 只看最终输出）。
- 通过返回 `Command` 可同时更新主 Agent 的状态键，携带结构化结果。

## 5. 检查点与状态

- 默认无状态：每次调用子代理都从初始状态开始，支持并发执行且互不干扰。
- 需要跨调用记忆时，使用 `checkpointer=True` 编译子代理，使其保持自己的对话历史（每线程模式）。注意此时并行调用会冲突。
- 子代理在工具内部被调用，因此无法通过 `get_state(subgraphs=True)` 查看其内部状态；如需可见性，应将子代理放在自定义图的节点中。

---

## 6. 与全局概念的关联

- **多智能体系统**：子代理是五种核心模式之一，提供最强的上下文隔离和集中控制。
- **工具 (Tools)**：子代理被封装为工具，与普通工具一样被主 Agent 调用。
- **子图 (Subgraphs)**：每个子代理通常是一个独立的 LangGraph 图；通过编译时的 checkpointer 控制其持久化策略。
- **上下文工程**：名称、描述、输入输出设计均属于上下文工程范畴，直接影响主 Agent 的决策质量和子代理的执行效果。
- **检查点 (Checkpointer)**：控制子代理是有状态还是无状态，决定了跨调用记忆和并发行为。
- **中断 (Interrupts)**：子代理内部可使用 `interrupt()` 暂停执行以收集用户输入，主 Agent 仍担任协调者。
- **性能**：子代理的同步/异步选择影响整体延迟；无状态设计保证每次调用成本一致但重复请求时无记忆优化。
