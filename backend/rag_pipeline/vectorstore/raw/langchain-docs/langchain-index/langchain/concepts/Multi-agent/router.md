# Router

> 这是 LangGraph 多智能体系统中 **Router（路由）** 模式的胖索引，覆盖路由分类、无状态与有状态实现、使用 `Command` 和 `Send` 的分发机制、与 Subagents 的对比以及上下文管理策略。
> 阅读本文档可一次性掌握路由模式的核心概念及其权衡，为构建面向垂直领域和并行查询的多智能体应用提供决策支撑。

---

## 概念全景

路由模式通过一个专门的分类步骤，将用户查询导向一个或多个垂直领域的专业智能体，再将结果综合为统一响应。它适合领域边界清晰、可并行处理的任务。

| 维度               | 描述                                                         |
| ------------------ | ------------------------------------------------------------ |
| **核心机制**       | 路由器对输入进行分类，然后通过 `Command` 或 `Send` 将查询分发给目标智能体 |
| **执行特征**       | 支持单智能体路由（`Command`）和并行扇出到多个智能体（`Send`） |
| **状态管理**       | 可无状态（每次请求独立路由）或有状态（维护对话历史）         |
| **综合步骤**       | 多个智能体的结果通常由一个综合节点合并成最终答案             |
| **与 Subagents 的区别** | 路由器是轻量级预处理步骤，不参与多轮编排；Subagents 由监督 Agent 动态决策 |

核心决策点：**选用无状态还是有状态路由、使用 `Command` 单路分发还是 `Send` 并行分发、是否将路由器包装为对话 Agent 的工具**。

---

## 1. 路由机制

### 使用 `Command` 单路分发

通过 `Command(goto=active_agent)` 将整个流程导向选定的专业智能体。适用于一次只应激活一个垂直领域的情形。

```python
def route_query(state: State) -> Command:
    active_agent = classify_query(state["query"])
    return Command(goto=active_agent)
```

### 使用 `Send` 并行扇出

当查询需要同时咨询多个领域时，路由函数返回 `Send` 对象列表，每个携带针对特定智能体的子查询，并行执行后聚合。

```python
def route_query(state: State):
    classifications = classify_query(state["query"])
    return [Send(c["agent"], {"query": c["query"]}) for c in classifications]
```

## 2. 无状态 vs 有状态

| 模式         | 特点                                                         | 适用场景                     |
| ------------ | ------------------------------------------------------------ | ---------------------------- |
| **无状态**   | 每个请求独立分类与路由，不保留对话历史。实现最简单，性能可预测。 | 一次性查询、无上下文依赖 |
| **有状态**   | 跨请求维护对话历史，以支持多轮交互。可通过工具包装或完全持久化实现。 | 多轮对话、需要上下文感知 |

### 无状态路由

适合将路由器作为预处理步骤，每次调用都重新分类。对于多轮对话，通常将无状态路由器包装为对话 Agent 的工具，由外部 Agent 维护历史。

### 有状态路由

两种实现方式：

- **工具包装器**：将路由器封装为对话 Agent 的工具，Agent 负责记忆和历史，路由器本身保持无状态。这是最简单且推荐的方式。
- **完全持久化**：路由器自身使用 checkpointer 存储历史，并有选择地向各个子智能体提供历史上下文。需要注意智能体间风格一致性，以及并行调用时的历史合并。在大多数情况下，建议优先考虑交接模式或子智能体模式来处理多轮需求。

## 3. Router vs Subagents

| 特性                 | Router                                      | Subagents (Supervisor)                       |
| -------------------- | ------------------------------------------- | -------------------------------------------- |
| **路由决策**         | 专用分类步骤（LLM 调用或规则）              | 监督 Agent 在对话中动态决策                  |
| **上下文管理**       | 通常不维护对话历史                          | 监督 Agent 维护全局上下文                    |
| **多轮编排**         | 否（单次分派）                              | 是（可在多轮中反复调用不同子智能体）         |
| **适合场景**         | 领域边界清晰、输入分类明确                  | 灵活、需对话感知的复杂多步工作流             |

---

## 4. 上下文工程

- **分类准确性**：分类步骤（可以是 LLM 调用或规则）的质量直接影响路由效果。考虑对模糊查询进行改写或澄清。
- **子智能体提示**：每个垂直领域的智能体应只包含其领域知识，避免上下文膨胀。
- **综合步骤**：如果并行调用了多个智能体，需要一个综合节点来汇总结果，确保最终答案连贯一致。
- **有状态路由的历史管理**：如果自己管理状态，必须有策略地选择向各个子智能体提供多少历史，避免无关上下文降低性能。

---

## 5. 与全局概念的关联

- **多智能体系统 (Multi-agent)**：路由是五种核心模式之一，最适合领域边界清晰、需要并行处理的场景。
- **Subgraphs (子图)**：路由的目标智能体通常是独立的子图，可通过 `Command` 或 `Send` 调用。
- **工具 (Tools)**：将路由器包装为工具是有状态路由的推荐方式，使其可集成到对话 Agent 中。
- **检查点 (Checkpointer)**：有状态完全持久化方案需要 checkpointer 来存储对话历史。
- **上下文工程**：分类、提示设计和选择性历史传递都是上下文工程的直接应用。
- **性能**：路由器在单次和并行多领域任务中调用次数和 Token 使用都表现高效。
