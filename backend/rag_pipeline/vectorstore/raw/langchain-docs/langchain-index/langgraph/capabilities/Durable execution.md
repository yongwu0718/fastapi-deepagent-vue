# 持久执行 (Durable execution)

> 这是 LangGraph 中**持久执行**的胖索引，覆盖启用要求、确定性重放规则、持久化模式、任务封装、恢复起点、优雅关闭及最佳实践。
> 阅读本文档可一次性掌握持久执行的全部概念及其关联，为构建可暂停、可恢复、抗故障的长时间运行工作流提供决策支撑。

---

## 概念全景

持久执行让工作流在关键点保存进度，能够在暂停后（无论间隔多久）准确地从停止处恢复，无需重新执行已完成的步骤。它天然支撑人机协同、长时间运行任务及故障恢复。

| 维度               | 描述                                                         |
| ------------------ | ------------------------------------------------------------ |
| **实现基础**       | LangGraph 内置持久化层（checkpointer），每个超级步骤后自动保存状态 |
| **恢复机制**       | 通过 `thread_id` 标识执行历史，恢复时从检查点重放，而不是从代码中断行继续 |
| **关键要求**       | 配置 checkpointer；提供 `thread_id`；将非确定性/副作用操作封装在任务或节点中 |
| **持久化模式**     | `exit`（最快，仅退出时持久化）、`async`（异步持久化，平衡）、`sync`（同步持久化，最高保障） |
| **优雅关闭**       | `RunControl.request_drain()` 协作式停止，保存检查点，后续可恢复 |
| **与中断的关系**   | `interrupt()` 是暂停执行的一种方式；从故障恢复通过传入 `None` 输入实现 |

核心决策点：**哪些操作必须封装为任务、选用哪种持久化模式、如何设计幂等操作、是否启用优雅关闭、恢复时依赖何种任务粒度**。

---

## 1. 启用要求

- **持久化层**：编译图时必须提供 checkpointer（`InMemorySaver` 开发用，`PostgresSaver` 等生产用）。
- **线程标识**：调用时在 `config["configurable"]["thread_id"]` 中指定唯一 ID。
- **任务封装**：任何非确定性或副作用操作必须放在任务（`@task`）或独立的节点中，以保证重放一致性。

---

## 2. 确定性与一致性重放

恢复时工作流会从合适的起点**重新执行**，而非从代码暂停行继续。因此必须保证执行确定性：

- **避免重复工作**：若一个节点包含多个副作用操作，将每个操作封装为独立**任务**，恢复时任务结果从持久化层直接获取，不会重复执行。
- **封装非确定性**：随机数生成等操作放入任务或节点。
- **幂等副作用**：API 调用、文件写入等应设计为幂等；若任务失败后重试，可避免重复副作用。使用幂等键或检查现有结果。

---

## 3. 持久化模式

通过 `graph.stream(..., durability="sync")` 等方式指定：

| 模式     | 行为                                                         | 性能 | 持久性       |
| -------- | ------------------------------------------------------------ | ---- | ------------ |
| `"exit"` | 仅在成功退出、错误退出或人机中断时持久化；中间状态不保存，无法从进程崩溃恢复 | 最高 | 最低         |
| `"async"` | 执行下一步的同时异步持久化；极小概率丢失检查点                   | 良好 | 中等         |
| `"sync"` | 下一步开始前同步持久化，确保每步都写入检查点                   | 较低 | 最高         |

按业务风险选择：对金融交易等高保障场景选 `sync`；对对话交互可选 `async`。

---

## 4. 在节点中使用任务

若节点内部包含多个操作，可将每个操作转换为 `@task`，避免重构为多个节点。任务执行结果会被持久化，恢复时直接读取。

```python
@task
def _make_request(url: str):
    return requests.get(url).text[:100]

def call_api(state: State):
    requests = [_make_request(url) for url in state['urls']]
    results = [r.result() for r in requests]
    return {"results": results}
```

同样适用于 Functional API。

---

## 5. 恢复工作流

- **暂停恢复**：使用 `interrupt()` 主动暂停，通过 `Command(resume=...)` 恢复。
- **故障恢复**：异常发生后，使用**相同 `thread_id`** 传入 `None` 作为输入重新执行，图将从最后一个成功检查点重放。
- **恢复起点**：
  - StateGraph (Graph API)：从发生中断或失败的**节点**开头重新执行。
  - 若节点内部调用了子图，则父节点开头重放；子图内部从具体中断节点恢复。
  - Functional API：从 `entrypoint` 开头重放。

---

## 6. 优雅关闭

需要 `langgraph>=1.2`（alpha）。通过 `RunControl` 在超级步骤边界协作停止运行，保存可恢复检查点。

- 创建 `RunControl`，调用 `control.request_drain(reason)` 发起停止。
- 图执行会引发 `GraphDrained` 异常，可从外部捕获。
- 恢复时使用相同的 `thread_id` 调用 `graph.invoke(None, config)`。
- 节点内可通过 `runtime.drain_requested` 感知 drain 状态，提前做轻量化处理。
- 适用于处理 SIGTERM 信号：安装信号处理器调用 `request_drain("sigterm")`。

---

## 7. 关键约束与最佳实践

- **必须配置 checkpointer**，否则无法保存进度。
- **thread_id 作为持久光标**：恢复时务必使用与中断/失败时相同的 ID。
- **副作用操作粒度**：尽量一个任务一个副作用，既保证幂等又不重复执行。
- **幂等设计**：外部 API 调用应使用唯一幂等键；文件写入考虑检查现有状态。
- **持久化模式权衡**：高性能场景选 `exit` 或 `async`；关键数据完整性选 `sync`。
- **优雅关闭与超时结合**：可对 drain 等待设置硬性超时，避免无限挂起。
- **避免在 `interrupt` 之前执行非幂等操作**（参见中断规则）。

---

## 8. 与全局概念的关联

- **持久化 (Persistence)**：持久执行完全依赖 checkpointer 和 store；所有模式、重放规则均基于持久化层。
- **中断 (Interrupts)**：`interrupt()` 是暂停执行的手段，与持久执行紧密结合，通过保存检查点实现恢复。
- **人机协同 (Human-in-the-loop)**：持久执行确保了人工审查期间进度不丢失。
- **容错 (Fault tolerance)**：重试、超时和错误处理均在持久执行框架下工作，失败后可从检查点恢复。
- **任务 (Tasks)**：是封装副作用的最小单元，与 Functional API 和 StateGraph 的 `@task` 直接对应。
- **优雅关闭**：与持久执行配合，实现受控停机而不丢失工作。
- **流式传输**：优雅关闭不影响流的正常结束；drain 状态可通过 `runtime` 访问。
