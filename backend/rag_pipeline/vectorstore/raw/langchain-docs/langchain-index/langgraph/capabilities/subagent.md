# Subgraphs

> 这是 LangGraph 中**子图（Subgraphs）**的胖索引，覆盖子图通信模式、持久化策略（每次调用、每个线程、无状态）、状态查看、流式传输及最佳实践。
> 阅读本文档可一次性掌握子图领域的全部概念及其关联，为构建多智能体系统、复用图组件和实现分布式开发提供决策支撑。

---

## 概念全景

子图是作为另一个图中节点使用的图，是实现多智能体系统、图组件复用和团队并行开发的核心机制。父图与子图的交互由通信模式和持久化策略共同决定。

| 维度                 | 描述                                                         |
| -------------------- | ------------------------------------------------------------ |
| **通信模式**         | 通过包装函数调用（不同状态模式）或直接作为节点添加（共享状态键） |
| **持久化策略**       | 每次调用（Per‑invocation）、每个线程（Per‑thread）或完全无状态 |
| **中断支持**         | 有状态子图继承父图 checkpointer，支持 `interrupt()` 暂停/恢复 |
| **状态查看**         | `graph.get_state(config, subgraphs=True)` 可访问子图内部检查点 |
| **流式输出**         | `stream(..., subgraphs=True)` 可输出子图事件，通过 `ns` 字段区分来源 |
| **并发调用**         | Per‑invocation 模式支持并行调用同一子图；Per‑thread 模式需防止检查点命名空间冲突 |

核心决策点：**选用哪种通信模式以适应状态模式差异、为子图选择哪种持久化级别（是否需要跨调用记忆或中断）、如何保证多个有状态子图的命名空间隔离**。

## 1. 子图通信模式

| 模式                       | 适用场景                                                     | 实现方式                                                     |
| -------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **在节点内部调用子图**     | 父图与子图状态模式不同（无共享键），或需要在调用前后转换状态 | 编写包装函数，将父状态映射为子图输入，将子图输出映射回父状态 |
| **将子图添加为节点**       | 父图与子图共享部分状态键，子图可直接读写父状态通道           | 直接将编译后的子图传递给 `add_node`，无需包装函数            |

- **包装函数调用**：适用于多智能体系统中每个智能体拥有私有消息历史的场景。子图完全封装自己的状态。
- **直接添加为节点**：适用于通过共享 `messages` 键进行通信的协作智能体，或任何需要与父图共享状态通道的场景。子图还可以拥有仅内部可见的私有状态键。

## 2. 子图持久化

通过 `.compile()` 的 `checkpointer` 参数控制持久化行为。父图必须本身使用 checkpointer 编译，子图持久化功能（中断、状态检查、每线程记忆）才会生效。

| 模式                 | `checkpointer=` | 行为                                                         |
| -------------------- | --------------- | ------------------------------------------------------------ |
| **每次调用（默认）** | `None`          | 每次调用都是全新的，单次调用内继承父 checkpointer，支持中断和持久执行。调用结束后状态丢弃。 |
| **每个线程**         | `True`          | 状态在同一线程的多次调用中累积，子图可从上次停止处继续执行。 |
| **无状态**           | `False`         | 没有 checkpointing，子图像普通函数调用，无法中断或持久执行。 |

### 每次调用（Per‑invocation）
- **适用场景**：大多数多智能体系统，子代理处理独立的一次性请求，无需记住之前的调用。
- **中断**：`interrupt()` 可暂停子图，`Command(resume=...)` 恢复。
- **并发**：同一父节点可以并行调用同一个子图多次，每次调用拥有独立的检查点命名空间，互不冲突。
- **状态不累积**：每次调用子图都从初始状态开始，不记得之前的对话。

### 每个线程（Per‑thread）
- **适用场景**：子代理需要跨多次调用累积上下文（如研究助手在几轮交流中构建知识）。
- **中断**：同样支持中断和恢复。
- **状态累积**：子图在同一线程的多次调用中保留消息历史和内部数据，每次调用从上一次状态继续。
- **并发限制**：Per‑thread 子图不支持并行调用同一个子图实例（会导致检查点命名空间冲突）。必须通过 `ToolCallLimitMiddleware` 或模型配置禁用并行工具调用。
- **命名空间隔离**：多个不同的 Per‑thread 子图（如水果专家和蔬菜专家）需要各自拥有唯一的命名空间，防止检查点相互覆盖。可通过将子代理包装在具有唯一节点名称的 `StateGraph` 中实现稳定的命名空间。

### 无状态（Stateless）
- **适用场景**：不需要 checkpointing 开销的纯函数式子图。
- **限制**：无法暂停/恢复，无法从持久执行中受益。进程崩溃后必须重头运行。

### 持久化特性对比

| 特性                           | 每次调用 | 每个线程 | 无状态 |
| ------------------------------ | -------- | -------- | ------ |
| 中断 (HITL)                    | ✅        | ✅        | ❌      |
| 跨调用记忆                     | ❌        | ✅        | ❌      |
| 同一节点多次调用同一子图       | ✅        | ❌        | ✅      |
| 同一节点调用不同子图           | ✅        | ⚠️ 需隔离 | ✅      |
| 状态查看 (`subgraphs=True`)    | ⚠️ 仅当前调用 | ✅        | ❌      |

## 3. 查看子图状态

- 通过 `graph.get_state(config, subgraphs=True)` 可获取当前任务中子图的 `StateSnapshot`。
- 每次调用模式：返回**仅当前调用**的子图状态。
- 每个线程模式：返回**该线程所有调用累积**的子图状态。
- 仅当子图被添加为节点或在节点内被静态调用时，状态才可被发现；在工具函数内部间接调用子图时，状态查看不可用（但中断仍能传播）。

---

## 4. 流式传输子图输出

- 在父图的 `stream()` 中设置 `subgraphs=True`，即可在输出中包含子图事件。
- 使用 `version="v2"` 时，子图事件遵循相同的 `StreamPart` 格式，`chunk["ns"]` 字段标识来源（根图为空元组，子图为 `('node_name:uuid',)`）。
- 无论子图嵌套多深，事件均按层次输出，便于前端按来源拆分渲染。

---

## 5. 关键约束与最佳实践

- **持久化模式选择**：大多数场景使用默认的每次调用模式即可；只有在子代理需要多轮对话记忆时才使用每线程模式，并注意处理并发调用限制和命名空间隔离。
- **并发调用**：Per‑thread 子图必须防止并行调用同一工具；使用 `ToolCallLimitMiddleware` 或禁用模型的并行工具调用。
- **命名空间隔离**：多个 Per‑thread 子图共存时，为每个子图提供唯一的包装节点名称，确保检查点命名空间稳定且不重叠。
- **中断传播**：`interrupt()` 在子图中被触发时，中断会向上冒泡到父图，父级流中通过 `__interrupt__` 捕获，恢复时需使用相同的 `thread_id`。
- **无状态子图的风险**：崩溃后无法恢复，仅适用于纯计算或幂等操作。
- **静态发现限制**：状态查看功能要求子图被直接添加为节点或在节点函数内静态调用；若在工具中动态调用，请勿依赖 `get_state(subgraphs=True)`。

---

## 6. 与全局概念的关联

- **多智能体系统**：子图是实现分层智能体的基础，每个子智能体可拥有独立的工具、记忆和中断逻辑。
- **持久化 (Persistence)**：子图持久化依赖父图 checkpointer；理解检查点命名空间是处理并发和恢复的关键。
- **中断 (Interrupts)**：子图中断自动传播到父图，使人机协同工作流可穿透任意深度的图层次。
- **流式传输 (Streaming)**：`subgraphs=True` 让子图内部事件对前端可见，适合构建可观察的多智能体交互界面。
- **记忆 (Memory)**：Per‑thread 模式为子图提供了跨调用的短期记忆；长期记忆仍通过 Store 在节点内共享。
- **上下文工程**：子图封装了特定的工具和提示，是模块化上下文管理的体现。
