# 优化技能描述

> 如何改进技能的描述，使其在相关的提示词下能被可靠地触发。

技能只有在被激活时才有用。`SKILL.md` 前言中的 `description` 字段，是agent用来决定是否为给定任务加载技能的主要机制。描述不充分，会导致技能在该触发时不触发；描述过于宽泛，则会导致在不该触发时触发。

本指南涵盖如何系统地测试和改进你的技能描述，以提高触发的准确性。

## 技能触发的工作原理

agent使用渐进式披露来管理上下文。在启动时，它们只加载每个可用技能的 `name` 和 `description`——仅够用于判断技能何时可能相关。当用户的任务与某个描述匹配时，agent会将完整的 `SKILL.md` 读入上下文，并遵循其中的指令。

这意味着描述承担着触发技能的全部责任。如果描述没有传达出技能何时有用，agent就不知道该去调用它。

一个重要的细微之处：agent通常只会在任务需要超出其独立处理能力的知识或能力时，才会查询技能。一个简单、一步到位的请求，比如“读一下这个 PDF”，即使描述完全匹配，也可能不会触发 PDF 技能，因为agent用基础工具就能处理。而那些涉及专业知识（如陌生的 API、特定领域的工作流或不常见的格式）的任务，才是写得好的描述能够发挥作用的地方。

## 编写有效的描述

在测试之前，了解一个好的描述是什么样子的会很有帮助。几条原则：

* **使用祈使句式。** 将描述写为对agent的指令：“当……时使用此技能”，而不是“此技能用于……”。agent正在决定是否要行动，所以要告诉它何时行动。
* **关注用户意图，而非实现方式。** 描述用户试图达成什么目标，而不是技能的内部机制。agent是根据用户的要求来进行匹配的。
* **宁可更主动一些。** 明确列出技能适用的上下文，包括用户没有直接提及该领域的场景：“即使用户没有明确提到‘CSV’或‘分析’。”
* **保持简洁。** 通常三两句话到一小段为宜——既要足够长以涵盖技能的范围，又要足够短以避免在许多技能中占用agent过多的上下文。根据 规范 规定，最大字符数限制为 1024 个字符。

## 设计触发评估查询

要测试触发情况，你需要一组评估查询——即标注了是否应触发你技能的、真实的用户提示。

```json
[
  { "query": "我有一张电子表格在 ~/data/q4_results.xlsx 里，C 列是收入，D 列是支出——你能帮我加一列利润率，并把低于 10% 的标记出来吗？", "should_trigger": true },
  { "query": "把这个 json 文件转成 yaml 的最快方法是什么", "should_trigger": false }
]
```

目标大约是 20 个查询：8-10 个应该触发的，8-10 个不该触发的。

### 应该触发的查询

这些查询测试描述是否涵盖了技能的范围。让它们在以下几个方面有所变化：

* **措辞**：有些正式，有些随意，有些带错别字或缩写。
* **明确性**：有些直接点出技能的领域（“分析这个 CSV”），另一些则描述需求而不指名（“我老板想让我从这个数据文件做个图表”）。
* **详细程度**：混合简短提示和上下文丰富的提示——一个简短的“分析我的销售 CSV 并做个图表”，与一个包含文件路径、列名和背景故事的长消息。
* **复杂度**：改变步骤数量和决策点。将单步任务与多步工作流结合起来，以测试当技能所要解决的任务被埋没在一个更大的任务链中时，agent是否仍能识别出它的相关性。

最有价值的应触发查询，是那些技能能帮上忙，但仅凭查询本身又无法明显看出关联的查询。这些正是描述措辞能起决定性作用的场景——如果查询已经明确要求了技能所提供的功能，那么任何合理的描述都会被触发。

### 不应该触发的查询

最有价值的负面测试案例是**近似匹配**——那些与你的技能共享关键词或概念，但实际上需要的是不同东西的查询。这些测试可以检验描述的精确性，而不仅仅是其广度。

对于一个 CSV 分析技能，较差的负面例子是：

* “写一个斐波那契函数” —— 明显不相关，什么都测试不了。
* “今天天气怎么样？”—— 没有关键词重叠，太简单了。

好的负面例子：

* “我需要更新我 Excel 预算电子表格里的公式” —— 共享了“电子表格”和“数据”的概念，但需要的是 Excel 编辑，而非 CSV 分析。
* “你能写个 Python 脚本读取 csv 并把每一行上传到我们的 postgres 数据库吗？”—— 涉及 CSV，但任务是数据库 ETL，而不是分析。

### 提升真实性的建议

真实的用户提示包含通用测试查询所缺乏的上下文。包含：

* 文件路径（`~/Downloads/report_final_v2.xlsx`）
* 个人背景（`“我的经理让我……”`）
* 具体细节（列名、公司名、数据值）
* 随意的语言、缩写和偶尔的错别字

## 测试一个描述是否会被触发

基本方法是：在安装了该技能的情况下，用agent运行每个查询，观察agent是否调用了该技能。确保技能已注册并可被你的agent发现——具体方式因客户端而异（例如，技能目录、配置文件或 CLI 标志）。

大多数agent客户端都提供某种形式的可观测性——执行日志、工具调用历史记录或详细输出——让你能够看到在运行过程中查询了哪些技能。详情请查看你的客户端文档。如果agent加载了你技能的 `SKILL.md`，则技能被触发；如果agent没有查阅它就继续执行，则没有触发。

一个查询“通过”的条件是：

* `should_trigger` 为 `true` 且技能被调用，或者
* `should_trigger` 为 `false` 且技能未被调用。

### 多次运行

模型行为是非确定性的——同一个查询可能这次触发了技能，下次却没有。对每个查询多次运行（3 次是个合理的起点），并计算**触发率**：技能被调用的次数占总运行次数的比例。

如果一个应触发查询的触发率高于某个阈值（0.5 是一个合理的默认值），则通过。如果一个不应触发查询的触发率低于该阈值，则通过。

20 个查询，每个运行 3 次，总共就是 60 次调用。你会需要脚本化这个过程。下面是通用结构——请将 `claude` 的调用和 `check_triggered` 中的检测逻辑替换为你agent客户端提供的相应功能：

```bash
#!/bin/bash
QUERIES_FILE="${1:?用法: $0 <queries.json>}"
SKILL_NAME="my-skill"
RUNS=3

# 此示例使用 Claude Code 的 JSON 输出来检查 Skill 工具调用。
# 请将此函数替换为你agent客户端的检测逻辑。
# 如果技能被调用，应返回 0（成功），否则返回 1。
check_triggered() {
  local query="$1"
  claude -p "$query" --output-format json 2>/dev/null \
    | jq -e --arg skill "$SKILL_NAME" \
      'any(.messages[].content[]; .type == "tool_use" and .name == "Skill" and .input.skill == $skill)' \
      > /dev/null 2>&1
}

count=$(jq length "$QUERIES_FILE")
for i in $(seq 0 $((count - 1))); do
  query=$(jq -r ".[$i].query" "$QUERIES_FILE")
  should_trigger=$(jq -r ".[$i].should_trigger" "$QUERIES_FILE")
  triggers=0

  for run in $(seq 1 $RUNS); do
    check_triggered "$query" && triggers=$((triggers + 1))
  done

  jq -n \
    --arg query "$query" \
    --argjson should_trigger "$should_trigger" \
    --argjson triggers "$triggers" \
    --argjson runs "$RUNS" \
    '{query: $query, should_trigger: $should_trigger, triggers: $triggers, runs: $runs, trigger_rate: ($triggers / $runs)}'
done | jq -s '.'
```

<Tip>
  如果你的agent客户端支持，你可以在结果明朗时提前终止运行——agent要么查阅了技能，要么已经开始在没有技能的情况下工作。这可以显著减少运行完整评估集所需的时间和成本。
</Tip>

## 使用训练/验证集划分避免过拟合

如果你针对所有查询来优化描述，就有过拟合的风险——制作出一个只对这些特定措辞有效，但对新措辞无效的描述。

解决方案是划分你的查询集：

* **训练集 (\~60%)**：用于发现失败案例并指导改进的查询。
* **验证集 (\~40%)**：留出的查询，仅用于检查改进是否具有泛化性。

确保两个集合都按比例包含应触发和不应触发的查询——不要意外地把所有正面案例都放进一个集合。随机打乱顺序，并在迭代过程中保持划分不变，这样才能进行同类比较。

如果你使用[上面](#running-multiple-times)那样的脚本，可以将查询拆分成两个文件——`train_queries.json` 和 `validation_queries.json`——并分别对每个文件运行脚本。

## 优化循环

1.  **评估** 当前描述在*训练集和验证集*上的表现。训练集的结果指导你的修改；验证集的结果告诉你这些修改是否具有泛化性。
2.  **识别失败案例** 在*训练集*中：哪些应触发的查询没有触发？哪些不应触发的查询触发了？
    * 仅使用训练集的失败案例来指导你的修改——无论你是自己修改描述还是用大模型生成，都要把验证集的结果排除在这个过程之外。
3.  **修改描述。** 专注于泛化改进：
    * 如果应触发的查询失败，可能是描述太窄了。扩大范围，或添加关于技能何时有用的上下文。
    * 如果不应触发的查询被误触发，可能是描述太宽泛了。添加说明，明确指出该技能*不*做什么，或澄清此技能与相邻功能之间的界限。
    * 避免直接从失败的查询中添加特定关键词——那是过拟合。相反，要找到这些查询所代表的通用类别或概念，并针对这个来解决。
    * 如果经过多次迭代后仍陷入困境，尝试采用结构上不同的方法来写描述，而不是渐进式的微调。一种不同的框架或句子结构也许能在改进无效的地方取得突破。
    * 检查描述是否保持在 1024 个字符的限制以内——在优化过程中，描述有不断增长的趋势。
4.  **重复**步骤 1-3，直到所有*训练集*查询都通过，或者你不再看到有意义的改进。
5.  **选择最佳迭代** 根据其在*验证集*上的通过率——验证集中查询通过的比例。请注意，最佳描述可能不是你生产的最后一个；较早的迭代版本可能比较晚的、已对训练集过拟合的版本有更高的验证通过率。

通常五次迭代就足够了。如果性能没有提高，问题可能出在查询上（太简单、太困难或标注不当），而不是描述本身。

<Tip>
  [`skill-creator`](https://github.com/anthropics/skills/tree/main/skills/skill-creator) 技能可以端到端地自动化这个循环：它拆分评估集，并行评估触发率，使用 Claude 提出描述改进建议，并生成一个可在运行时查看的实时 HTML 报告。
</Tip>

## 应用结果

一旦你选定了最佳描述：

1.  更新 `SKILL.md` 前言中的 `description` 字段。
2.  验证描述是否在 [1024 字符限制](/specification#description-field) 以内。
3.  验证描述是否能如预期般触发。手动尝试几个提示作为一个快速的合理性检查。要进行更严格的测试，编写 5-10 个全新的查询（混合应触发和不应触发的），并用评估脚本运行它们——由于这些查询从未参与优化过程，它们能诚实地检验描述是否具有泛化性。

优化前后的对比：

```yaml theme={null}
# 之前
description: 处理 CSV 文件。

# 之后
description: >
  分析 CSV 和表格数据文件——计算汇总统计、添加派生列、生成图表并清理脏数据。
  当用户有 CSV、TSV 或 Excel 文件并想要探索、转换或可视化数据时，使用此技能，
  即使用户没有明确提到 "CSV" 或 "分析"。
```

改进后的描述更具体地说明了技能的功能（汇总统计、派生列、图表、数据清理），并更广泛地说明了其适用范围（CSV、TSV、Excel；即使没有明确的关键词）。
