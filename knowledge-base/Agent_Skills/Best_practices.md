# skill创建者的最佳实践

> 如何编写范围得当、与任务相匹配的skill。
## 从真实的专业经验出发

skill创建中一个常见的陷阱是，仅让大语言模型（LLM）生成skill，而不提供特定领域的上下文——完全依赖大模型通用的训练知识。结果产生的往往是模糊、泛泛的操作流程（比如“适当处理错误”、“遵循身份验证的最佳实践”），而非那些能让skill真正有价值的特定 API 模式、边界情况和项目约定。

**有效的skill应扎根于真实的专业经验。关键在于将特定领域的上下文输入到创建过程中。**

### 从动手实操的任务中提炼

在与agent对话的过程中完成一项真实任务，全程提供上下文、纠正和偏好。然后将可复用的模式提炼成skill。注意以下几点：

* **哪些步骤有效** —— 导向成功的一系列操作
* **你做了哪些纠正** —— 你引导agent调整方法的地方（例如，“用库 X 而不是 Y”，“检查边界情况 Z”）
* **输入/输出格式** —— 数据输入和输出时的形态
* **你提供了哪些上下文** —— agent此前不知道的、项目特有的事实、约定或约束

### 从现有的项目资料中综合

当你拥有一批现成的知识时，可以将它们输入大模型，让模型综合出一个skill。根据团队真实的事故报告和运维手册综合出的数据流水线skill，其效果将远超根据一篇泛泛的“数据工程最佳实践”文章生成的skill，因为它捕捉到了*你们*的数据模式、故障模式和恢复流程。关键在于使用项目特有的材料，而非通用参考资料。

好的素材来源包括：

* 内部文档、运维手册和风格指南
* API 规范、数据库模式和配置文件
* 代码评审意见和问题跟踪器（能捕捉到反复出现的问题和评审者的期望）
* 版本控制历史，特别是补丁和修复（通过实际发生的变更揭示模式）
* 真实的故障案例及其解决方法

## 通过实际执行来改进

skill的初稿通常需要改进。用真实的 tasks 来运行这个skill，然后将运行结果——全部结果，而不仅仅是失败的那部分——反馈到创建过程中。自问：什么触发了误报？遗漏了什么？哪些内容可以删减？

哪怕只做一轮“执行-修订”的过程，也能显著提升质量。复杂的领域通常需要多轮改进才会更好。

<Tip>
  要阅读agent的执行轨迹，而不只是看最终的输出。如果agent把时间浪费在了无效的步骤上，常见原因包括：指令太模糊（agent会尝试多种方法才找到可行的那个）、指令与当前任务无关（agent依然会照做），或者提供了太多选项却没有给出明确的默认选择。
</Tip>

关于更结构化的迭代方法，包括测试用例、断言和评分，请参见[评估skill输出质量](Evaluating-skill-output-quality.md)。

## 明智地消耗上下文

一旦某个skill被激活，其完整的 `SKILL.md` 内容就会加载到agent的上下文窗口中，与对话历史、系统上下文以及其他已激活的skill共存。skill中的每个 token 都在与窗口中的其他一切争夺agent的注意力。

### 添加agent缺少的，省去它已知的

专注于那些没有你的skill，agent*就不会知道*的内容：项目特定的约定、领域特定的流程、非显而易见的边界情况，以及要使用的特定工具或 API。你不需要解释什么是 PDF、HTTP 如何工作，或者数据库迁移是做什么的。

````markdown
<!-- 太过啰嗦 —— agent已经知道 PDF 是什么 -->
## 提取 PDF 文本

PDF（便携式文档格式）是一种常见的文件格式，包含文本、图片和其他内容。
要从 PDF 中提取文本，你需要使用一个库。推荐使用 pdfplumber，因为它能很好地处理大多数情况。

<!-- 更好的写法 —— 直接切入agent自己不知道的内容 -->
## 提取 PDF 文本

使用 pdfplumber 进行文本提取。对于扫描件，回退使用 pdf2image 配合 pytesseract。

```python
import pdfplumber

with pdfplumber.open("file.pdf") as pdf:
    text = pdf.pages[0].extract_text()
```
````

针对每条内容问问自己：“如果没有这条指令，agent会做错吗？” 如果答案是否定的，就删掉它。如果不确定，测试一下。如果agent已经能在没有该skill的情况下很好地完成整个任务，这个skill可能就没有增加价值。关于如何系统地进行测试，请参阅[评估skill输出质量](/skill-creation/evaluating-skills)。

### 设计内聚的单元

决定一个skill应该涵盖什么内容，就像决定一个函数应该做什么一样：你希望它封装一个内聚的工作单元，并能与其他skill很好地组合。范围太窄的skill会迫使处理单个任务时需要加载多个skill，从而带来开销和指令冲突的风险。范围太宽的skill则难以被精确激活。一个用于查询数据库并格式化结果的skill可能就是一个内聚的单元，而一个同时涵盖数据库管理功能的skill就可能试图做太多事情了。

### 追求适度的详细程度

过于全面的skill弊大于利——agent难以提取出相关的内容，并可能因为某些不适用于当前任务的指令而走上无用的路径。简洁、分步的指导加上一个可工作的示例，效果往往优于详尽的文档。当你发现自己试图覆盖所有边界情况时，请考虑大部分情况是否可以交给agent自己的判断来处理。

### 使用渐进式披露构建大型skill

规范建议将 `SKILL.md` 控制在 500 行和 5,000 个 token 以内——仅包含agent每次运行时所需的核心指令。当skill确实需要更多内容时，将详细的参考资料移到 `references/` 或类似目录下的独立文件中。

关键是告诉agent*何时*加载每个文件。“如果 API 返回非 200 状态码，请阅读 `references/api-errors.md`”比泛泛的“详情参见 references/”更有用。这样agent可以按需加载上下文，而非一开始就全部加载，这正是渐进式披露的设计初衷。

## 校准控制粒度

并非skill的每个部分都需要相同程度的指令明确性。要根据任务的脆弱程度来匹配指令的具体程度。

### 将明确性与脆弱性相匹配

**给agent自由度**，当多种方法都有效且任务能容忍变化时。对于灵活的指令，解释*为什么*可能比僵化的指令更有效——理解指令背后目的的agent能做出更好的上下文相关决策。一个代码审查skill可以描述要检查什么，而不必规定确切的步骤：

```markdown
## 代码审查流程

1. 检查所有数据库查询是否存在 SQL 注入漏洞（应使用参数化查询）
2. 验证每个端点都进行了身份验证检查
3. 寻找并发代码路径中的竞态条件
4. 确认错误信息不会泄露内部细节
```

**要给出明确的指示**，当操作很脆弱、一致性至关重要或必须遵循特定顺序时：

````markdown
## 数据库迁移

严格按照此顺序运行：

```bash
python scripts/migrate.py --verify --backup
```

不要修改该命令，也不要添加额外的标志。
````

大多数skill都是混合型的。请独立校准每个部分。

### 提供默认方案，而非选项菜单

当多种工具或方法都可行时，选择一个作为默认方案，并简要提及其他替代选项，而不是将它们作为并列选项呈现。

````markdown
<!-- 选项太多 -->
你可以使用 pypdf、pdfplumber、PyMuPDF 或 pdf2image...

<!-- 清晰的默认选项，并留有退路 -->
使用 pdfplumber 进行文本提取：

```python
import pdfplumber
```

对于需要 OCR 的扫描版 PDF，改用 pdf2image 配合 pytesseract。
````

### 倾向于提供流程而非声明

一个skill应该教会agent*如何处理*一类问题，而不是针对某个特定实例*产出什么*。对比一下：

```markdown
<!-- 具体答案 —— 仅对这个确切的任务有用 -->
将 `orders` 表与 `customers` 表在 `customer_id` 上连接，筛选出 `region = 'EMEA'` 的记录，并对 `amount` 列求和。

<!-- 可复用的方法 —— 适用于任何分析查询 -->
1. 从 `references/schema.yaml` 中读取 schema，找到相关表
2. 使用 `_id` 外键命名约定来连接表
3. 将用户请求中的任何筛选条件作为 WHERE 子句应用
4. 根据需要聚合数值列，并将结果格式化为 markdown 表格
```

这并不意味着skill不能包含具体细节——输出格式模板（参见[输出格式模板](#输出格式模板)）、诸如“绝不输出个人身份信息（PII）”的约束，以及针对特定工具的说明，这些都是有价值的。关键在于其*方法*应该具有通用性，即使个别细节很具体。

## 有效指令的模式

这些是构建skill内容的可复用技巧。不是每个skill都需要全部用到——请选用适合你任务的那些。

### “注意事项”小节

许多skill中价值最高的内容就是一份注意事项清单——那些与合理假设相悖、与特定环境相关的事实。它们不是泛泛的建议（比如“适当处理错误”），而是对agent在无人明确告知的情况下会犯的错的具体纠正：

```markdown
## 注意事项

- `users` 表使用了软删除。查询必须包含 `WHERE deleted_at IS NULL`，否则结果中将包含已停用的账户。
- 用户 ID 在数据库中是 `user_id`，在认证服务中是 `uid`，在账单 API 中是 `accountId`。三者指向同一个值。
- `/health` 端点只要 Web 服务器在运行就返回 200，即使数据库连接已断开也不例外。请使用 `/ready` 来检查完整的服务健康状况。
```

请将注意事项放在 `SKILL.md` 中，确保agent在遇到相关情况前能读到它们。如果使用独立的参考资料文件，虽然也可以，但必须告诉agent何时去加载它。然而对于非显而易见的陷阱，agent可能识别不出加载的触发条件。

***
当agent犯了一个你必须纠正的错误时，就把这个纠正加入注意事项小节。这是迭代改进skill的最直接方式之一（参见通过实际执行来改进）。
***

### 输出格式模板

当你需要agent以特定格式生成输出时，提供一个模板。这比用文字描述格式更可靠，因为agent非常擅长将具体结构进行模式匹配。较短的模板可以直接放在 `SKILL.md` 中；对于较长的模板，或仅在特定情况下才需要的模板，可将它们存放在 `assets/` 目录下，并在 `SKILL.md` 中引用，以便它们仅在需要时才加载。

````markdown
## 报告结构

使用此模板，根据具体分析需要调整各部分：

```markdown
# [分析标题]

## 执行摘要
[一段话概述关键发现]

## 关键发现
- 发现一，附支持数据
- 发现二，附支持数据

## 建议
1. 具体可行的建议
2. 具体可行的建议
```
````

### 用于多步骤工作流的清单

明确的清单有助于agent跟踪进度，避免遗漏步骤，特别是当步骤之间存在依赖关系或需要验证节点时。

```markdown theme={null}
## 表单处理工作流

进度：
- [ ] 步骤 1：分析表单（运行 `scripts/analyze_form.py`）
- [ ] 步骤 2：创建字段映射（编辑 `fields.json`）
- [ ] 步骤 3：验证映射（运行 `scripts/validate_fields.py`）
- [ ] 步骤 4：填写表单（运行 `scripts/fill_form.py`）
- [ ] 步骤 5：校验输出（运行 `scripts/verify_output.py`）
```

### 验证循环

指示agent在继续之前验证自己的工作。其模式是：完成工作，运行验证器（一个脚本、一份参考清单或一次自检），修复发现的问题，重复此过程直到验证通过。

```markdown theme={null}
## 编辑工作流

1. 进行编辑
2. 运行验证：`python scripts/validate.py output/`
3. 如果验证失败：
   - 查看错误信息
   - 修复问题
   - 再次运行验证
4. 仅在验证通过后才继续
```

一份参考文档也可以充当“验证器”——指示agent在定稿前对照参考资料检查其工作。

### 计划-验证-执行

对于批处理或破坏性操作，让agent以结构化格式创建中间计划，依据单一事实来源进行验证，然后才执行。

```markdown
## PDF 表单填写

1. 提取表单字段：`python scripts/analyze_form.py input.pdf` → `form_fields.json`
   （列出每个字段的名称、类型以及是否必填）
2. 创建 `field_values.json`，将每个字段名映射到其预期值
3. 验证：`python scripts/validate_fields.py form_fields.json field_values.json`
   （检查每个字段名在表单中是否存在，类型是否兼容，且必填字段没有缺失）
4. 如果验证失败，修改 `field_values.json` 并重新验证
5. 填写表单：`python scripts/fill_form.py input.pdf field_values.json output.pdf`
```

关键要素是第 3 步：一个验证脚本，它将计划（`field_values.json`）与事实来源（`form_fields.json`）进行核对。像“字段 'signature_date' 未找到 —— 可用字段：customer_name, order_total, signature_date_signed”这样的错误信息，能给agent提供足够的信息进行自我纠正。

### 捆绑可复用的脚本

在迭代skill时，比较agent在多个测试用例中的执行轨迹。如果你发现agent在每次运行中都独立地重复造同样的轮子——例如构建图表、解析特定格式、验证输出——这就是一个信号：应该编写一个经过测试的脚本，并将其捆绑在 `scripts/` 目录中。

关于设计和捆绑脚本的更多内容，请参阅在skill中使用[脚本](uv-add-deepagents-acp.md)。
