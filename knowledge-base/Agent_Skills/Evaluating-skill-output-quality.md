# 评估技能输出质量

> 如何通过评估驱动的迭代来测试你的技能是否产出好的结果。

你写了一个技能，在一个提示上试了试，看起来能工作。但它是否可靠——在各种不同的提示下、在边界情况中、比完全没有技能更好吗？运行结构化的评估可以回答这些问题，并为你系统地改进技能提供一个反馈循环。

## 设计测试用例

一个测试用例包含三个部分：

* **提示（Prompt）**：一段真实的用户消息——用户实际会输入的那种内容。
* **预期输出（Expected output）**：一段人类可读的、描述成功样子的说明。
* **输入文件（Input files）**（可选）：技能运行所需的文件。

将测试用例保存在技能目录下的 `evals/evals.json` 中：

```json
{
  "skill_name": "csv-analyzer",
  "evals": [
    {
      "id": 1,
      "prompt": "我有一份月度销售数据的 CSV 文件在 data/sales_2025.csv 中。你能找出收入最高的 3 个月并制作一个柱状图吗？",
      "expected_output": "一张柱状图图片，展示收入最高的 3 个月，并带有轴标签和数值。",
      "files": ["evals/files/sales_2025.csv"]
    },
    {
      "id": 2,
      "prompt": "我下载文件夹里有个叫 customers.csv 的文件，有些行缺了邮箱——你能清理一下并告诉我缺了多少吗？",
      "expected_output": "一份处理了缺失邮箱的、清理后的 CSV 文件，以及缺失数量的统计。",
      "files": ["evals/files/customers.csv"]
    }
  ]
}
```

**编写好的测试提示的建议：**

* **从 2-3 个测试用例开始。** 在拿到第一轮结果之前，不要过度投入。之后你可以扩展测试集。
* **让提示多样化。** 使用不同的措辞、详细程度和正式程度。有些提示可以很随意（“嘿，你能清理一下这个 csv 吗”），另一些则要精确（“解析 data/input.csv 处的 CSV，删除 B 列为空的行，并将结果写入 data/output.csv”）。
* **覆盖边界情况。** 至少包含一个测试边界条件的提示——一个格式错误的输入、一个不寻常的请求，或者技能指令可能模棱两可的情况。
* **使用真实的上下文。** 真实用户会提到文件路径、列名和个人背景。像“处理这个数据”这样的提示太模糊，无法测试任何有用的东西。

现在还不用急于定义具体的通过/失败检查——只需要提示和预期输出。在看完第一次运行产生的结果后，你会添加详细的检查（被称为断言）。

## 运行评估

核心模式是每个测试用例运行两次：一次**有技能**，一次**没有技能**（或使用之前的版本）。这为你提供了一个用于比较的基线。

### 工作空间结构

将评估结果组织在工作空间目录中，与技能目录并列。每一次完整的评估循环都在各自的 `iteration-N/` 目录中进行。在其中，每个测试用例都有一个评估目录，内含 `with_skill/` 和 `without_skill/` 子目录：

```
csv-analyzer/
├── SKILL.md
└── evals/
    └── evals.json
csv-analyzer-workspace/
└── iteration-1/
    ├── eval-top-months-chart/
    │   ├── with_skill/
    │   │   ├── outputs/       # 运行产生的文件
    │   │   ├── timing.json    # Token 数和耗时
    │   │   └── grading.json   # 断言结果
    │   └── without_skill/
    │       ├── outputs/
    │       ├── timing.json
    │       └── grading.json
    ├── eval-clean-missing-emails/
    │   ├── with_skill/
    │   │   ├── outputs/
    │   │   ├── timing.json
    │   │   └── grading.json
    │   └── without_skill/
    │       ├── outputs/
    │       ├── timing.json
    │       └── grading.json
    └── benchmark.json         # 汇总统计数据
```

你手动编写的主要文件是 `evals/evals.json`。其他的 JSON 文件（`grading.json`、`timing.json`、`benchmark.json`）是在评估过程中由代理、脚本或你来生成的。

### 启动运行

每次评估运行都应该从一个干净的上下文开始——不能有之前运行或技能开发过程中残留的状态。这确保了代理只遵循 `SKILL.md` 中的指示。在支持子代理的环境（例如 Claude Code）中，这种隔离是自然而然实现的：每个子任务都重新开始。如果没有子代理，则需要为每次运行使用单独的会话。

对于每次运行，提供：

* 技能路径（或基线运行时的无技能）

* 测试提示
* 任何输入文件
* 输出目录

以下是你为单次有技能的运行向代理下达指令的示例：

```
执行此任务：
- 技能路径：/path/to/csv-analyzer
- 任务：我有一份月度销售数据的 CSV 文件在 data/sales_2025.csv 中。
  你能找出收入最高的 3 个月并制作一个柱状图吗？
- 输入文件：evals/files/sales_2025.csv
- 将输出保存到：csv-analyzer-workspace/iteration-1/eval-top-months-chart/with_skill/outputs/
```

对于基线运行，使用相同的提示但不带技能路径，保存到 `without_skill/outputs/`。

当改进现有技能时，使用上一个版本作为基线。在编辑之前对其进行快照（`cp -r <skill-path> <workspace>/skill-snapshot/`），将基线运行指向该快照，并保存到 `old_skill/outputs/` 而非 `without_skill/`。

### 捕获时序数据

时序数据能让你比较技能相对于基线所花费的时间和 token 成本——一个能极大提升输出质量但将 token 用量增加两倍的技能，与一个既更好又更便宜的技能，代表着不同的取舍。每次运行完成后，记录 token 数量和耗时：

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332
}
```

<Tip>
  在 Claude Code 中，当子代理任务完成时，[任务完成通知](https://platform.claude.com/docs/en/agent-sdk/typescript#sdk-task-notification-message) 会包含 `total_tokens` 和 `duration_ms`。请立即保存这些值——它们不会持久化在其他地方。
</Tip>

## 编写断言

断言是关于输出应包含什么或应达到什么效果的可验证陈述。在你看到第一轮输出后再添加它们——通常在技能运行之前，你并不知道“好”是什么样子的。

好的断言：

* `"输出文件是有效的 JSON"` —— 可通过程序验证。
* `"柱状图有轴标签"` —— 具体且可观察。
* `"报告至少包含 3 条建议"` —— 可计数。

差的断言：

* `"输出很好"` —— 太过模糊，无法评分。
* `"输出完全使用了短语 'Total Revenue: $X'"` —— 太脆弱；正确的输出如果用词不同就会失败。

并非所有东西都需要断言。有些品质——如写作风格、视觉设计、输出是否“感觉对”——很难分解为通过/失败的检查。这些最好在[人类审查](#由人类审查结果)环节中捕捉。将断言留给那些可以客观检查的事情。

在 `evals/evals.json` 中为每个测试用例添加断言：

```json evals/evals.json highlight={9-14} theme={null}
{
  "skill_name": "csv-analyzer",
  "evals": [
    {
      "id": 1,
      "prompt": "我有一份月度销售数据的 CSV 文件在 data/sales_2025.csv 中。你能找出收入最高的 3 个月并制作一个柱状图吗？",
      "expected_output": "一张柱状图图片，展示收入最高的 3 个月，并带有轴标签和数值。",
      "files": ["evals/files/sales_2025.csv"],
      "assertions": [
        "输出包含一个柱状图图片文件",
        "图表恰好显示了 3 个月",
        "两个轴都有标签",
        "图表标题或说明提到了收入"
      ]
    }
  ]
}
```

## 对输出评分

评分意味着根据实际输出评估每项断言，并记录**通过（PASS）**或**失败（FAIL）**，附上具体的证据。证据应该引用或提到输出内容，而不只是陈述一个观点。

最简单的方法是将输出和断言交给一个大模型（LLM），要求它评估每一条。对于那些可以通过代码检查的断言（如有效的 JSON、正确的行数、文件存在且尺寸符合预期），使用验证脚本——对于机械性的检查，脚本比大模型的判断更可靠，并且可以在多次迭代中复用。

```json
{
  "assertion_results": [
    {
      "text": "输出包含一个柱状图图片文件",
      "passed": true,
      "evidence": "在输出目录中找到 chart.png (45KB)"
    },
    {
      "text": "图表恰好显示了 3 个月",
      "passed": true,
      "evidence": "图表显示了三月、七月和十一月的柱形"
    },
    {
      "text": "两个轴都有标签",
      "passed": false,
      "evidence": "Y 轴标记为 'Revenue ($)'，但 X 轴没有标签"
    },
    {
      "text": "图表标题或说明提到了收入",
      "passed": true,
      "evidence": "图表标题为 'Top 3 Months by Revenue'"
    }
  ],
  "summary": {
    "passed": 3,
    "failed": 1,
    "total": 4,
    "pass_rate": 0.75
  }
}
```

### 评分原则

* **要求提供具体的证据才能判为通过。** 不要给“疑点利益”。如果一条断言写着“包含一个摘要”，而输出中有一个标题为“摘要”的部分，里面只有一句含糊的话，那这就是失败——标签有了，但实质内容没有。
* **审查断言本身，而不仅仅是结果。** 在评分时，注意那些太简单（无论技能质量如何总是通过）、太难（即使输出很好也总是失败）或无法验证（无法仅从输出检查）的断言。在下一次迭代中修复它们。

<Tip>
  要比较两个技能版本，可以尝试**盲测对比**：将两个输出交给大模型裁判，但不透露哪个来自哪个版本。裁判根据自己的评分标准对整体质量——组织、格式、可用性、完善度——进行评分，而不带有哪个版本“应该”更好的偏见。这是对断言评分的补充：两个输出可能都通过了所有断言，但在整体质量上有显著差异。
</Tip>

## 汇总结果

一旦本次迭代中的每次运行都评完分，就计算每个配置的汇总统计信息，并将其保存到评估目录旁边的 `benchmark.json` 中（例如 `csv-analyzer-workspace/iteration-1/benchmark.json`）：

```json
{
  "run_summary": {
    "with_skill": {
      "pass_rate": { "mean": 0.83, "stddev": 0.06 },
      "time_seconds": { "mean": 45.0, "stddev": 12.0 },
      "tokens": { "mean": 3800, "stddev": 400 }
    },
    "without_skill": {
      "pass_rate": { "mean": 0.33, "stddev": 0.10 },
      "time_seconds": { "mean": 32.0, "stddev": 8.0 },
      "tokens": { "mean": 2100, "stddev": 300 }
    },
    "delta": {
      "pass_rate": 0.50,
      "time_seconds": 13.0,
      "tokens": 1700
    }
  }
}
```

`delta` 值告诉你技能的成本（更多时间，更多 token）和收益（更高的通过率）。一个增加了 13 秒但将通过率提高了 50 个百分点的技能，可能是值得的。一个将 token 用量翻倍而仅带来 2 个百分点提升的技能，则可能不值得。

***
  标准差（`stddev`）仅在每个评估有多次运行时才有意义。在早期迭代中，如果只有 2-3 个测试用例且每个只运行一次，请关注原始的通过数量和 delta 值——随着你扩大测试集并对每个评估运行多次，统计指标才会变得有用。
***

## 分析模式

汇总统计可能会隐藏重要的模式。在计算出基准数据后：

* **移除或替换在两种配置中都总是通过的断言。** 这些不能告诉你任何有用的信息——没有技能时模型也能很好地处理它们。它们会抬高有技能时的通过率，却不能反映技能的实际价值。
* **调查在两种配置中都总是失败的断言。** 要么断言本身有问题（要求了模型做不到的事情），要么测试用例太难，或者断言检查的对象不对。在下一次迭代前修复它们。
* **研究在有技能时通过、无技能时失败的断言。** 这正是技能明显在增加价值的地方。理解*原因*——是哪条指令或脚本造成了差异？
* **当多次运行结果不一致时，收紧指令。** 如果同一个评估有时通过有时失败（表现为基准数据中较高的 `stddev`），那么评估可能不可靠（对模型的随机性敏感），或者技能的指令可能足够含糊，导致模型每次的理解都不同。添加示例或更具体的指导来减少歧义。
* **检查时间和 token 的异常值。** 如果某个评估花费的时间是其他的 3 倍，请阅读它的执行记录（模型在运行期间所做操作的全部日志）以找到瓶颈所在。

## 由人类审查结果

断言评分和模式分析能发现很多问题，但它们只能检查你想到要写断言的那些方面。人类审查者能带来全新的视角——发现你未曾预料到的问题，注意到输出在技术上正确但没有抓住要点的情况，或者指出那些难以表达为通过/失败检查的问题。对于每个测试用例，审查实际的输出以及相应的评分。

为每个测试用例记录具体的反馈，并将其保存在工作空间中（例如，作为评估目录旁边的 `feedback.json` 文件）：

```json
{
  "eval-top-months-chart": "图表缺少轴标签，并且月份的排序是按字母顺序而不是按时间顺序。",
  "eval-clean-missing-emails": ""
}
```

“图表缺少轴标签”是可操作的反馈；而“看起来很差”则不是。空白的反馈意味着输出看起来没问题——那个测试用例通过了你的审查。在[迭代步骤](#迭代技能)中，将改进的重点放在那些你有具体意见的测试用例上。

## 迭代技能

在评分和审查之后，你有了三个信号来源：

* **失败的断言** 指向具体的差距——一个缺失的步骤、一条不清晰的指令，或者技能没有处理的一个案例。
* **人类的反馈** 指向更广泛的质量问题——方法错了，输出结构不佳，或者技能产生了技术上正确但无用的结果。
* **执行记录** 揭示了事情*为什么*出错。如果代理忽略了一条指令，那条指令可能含糊不清。如果代理在无效的步骤上花费了时间，那些指令可能需要简化或移除。

将这些信号转化为技能改进的最有效方法，是将这三者——连同当前的 `SKILL.md`——一起交给一个大模型，并要求它提出修改建议。大模型可以综合失败的断言、审查者的抱怨以及执行记录中的行为模式，这些模式如果靠人工连接会非常繁琐。在向大模型给出提示时，包含以下指导方针：

* **从反馈中进行泛化。** 该技能将用于许多不同的提示，而不仅仅是这些测试用例。修复应广泛地解决潜在的问题，而不是为特定的例子添加狭窄的补丁。
* **保持技能的精简。** 更少、更好的指令往往优于详尽的规则。如果执行记录显示有浪费的工作（不必要的验证，不需要的中间输出），就移除那些指令。如果尽管添加了更多规则，通过率仍然停滞不前，那么技能可能被过度约束了——尝试移除指令，看看结果是否能保持或改善。
* **解释为什么。** 基于推理的指令（“做 X，因为 Y 往往会导致 Z”）比死板的指令（“总是做 X，绝不做 Y”）效果更好。当模型理解其目的时，会更可靠地遵循指令。
* **捆绑重复性的工作。** 如果每次测试运行都独立编写了一个类似的辅助脚本（如图表生成器、数据解析器），这提示你应将脚本捆绑到技能的 `scripts/` 目录中。关于具体做法，参见[使用脚本](/skill-creation/using-scripts)。

### 循环

1. 将评估信号和当前的 `SKILL.md` 交给一个大模型，并要求它提出改进建议。
2. 审查并应用这些更改。
3. 在新的 `iteration-<N+1>/` 目录中重新运行所有测试用例。
4. 评分并汇总新的结果。
5. 由人类审查。重复此过程。

当你对结果感到满意，反馈持续为空，或者你在两次迭代之间不再看到有意义的改进时，就可以停止了。


  [`skill-creator`](https://github.com/anthropics/skills/tree/main/skills/skill-creator) 技能自动化了这个工作流的大部分——运行评估、给断言评分、汇总基准，以及展示结果供人类审查。
