# Running Containers（运行容器）

本文档整合了 Docker 运行容器的核心概念，涵盖端口发布、默认配置覆盖、文件共享、数据持久化和多容器应用五个主题。

---

## 一、Publishing and Exposing Ports（发布和暴露端口）

容器为应用的每个组件提供了隔离的进程。每个组件都运行在各自独立的沙箱环境中，与宿主机上的其他一切完全隔离。这种隔离意味着你无法直接访问它们——这就是 **发布端口（publishing ports）** 的作用所在。

### 1.1 Publishing Ports（发布端口）

发布端口通过设置转发规则，能够突破部分网络隔离。使用 `docker run` 命令的 `-p`（或 `--publish`）标志：

```console
docker run -d -p HOST_PORT:CONTAINER_PORT nginx
```

- `HOST_PORT`：宿主机上用于接收流量的端口号
- `CONTAINER_PORT`：容器内部监听的端口号

例如，将容器的 `80` 端口发布到宿主机的 `8080` 端口：

```console
docker run -d -p 8080:80 nginx
```

> **重要**：默认情况下，端口发布时会发布到所有网络接口。这意味着任何能访问你机器的流量都可以访问已发布的应用。

### 1.2 Publishing to Ephemeral Ports（发布到临时端口）

如果不想指定宿主机端口，可以让 Docker 自动选择（省略 `HOST_PORT`）：

```console
docker run -p 80 nginx
```

使用 `docker ps` 查看分配的临时端口：

```console
docker ps
# CONTAINER ID   IMAGE    ...   PORTS
# a527355c9c53   nginx    ...   0.0.0.0:54772->80/tcp
```

### 1.3 Publishing All Ports（发布所有端口）

使用 `-P` 或 `--publish-all` 标志自动将所有 `EXPOSE` 声明的端口发布到临时端口：

```console
docker run -P nginx
```

### 1.4 使用 Docker Compose 发布端口

在 `compose.yaml` 中配置端口映射：

```yaml
services:
  app:
    image: docker/welcome-to-docker
    ports:
      - 8080:80
```

然后使用 `docker compose up` 启动应用。

## 二、Overriding Container Defaults（覆盖容器默认配置）

Docker 容器从镜像配置中获取默认的可执行文件和设置，但你可以根据需要覆盖这些默认值。

### 2.1 Overriding Network Ports（覆盖网络端口）

将容器端口映射到不同的宿主机端口，避免冲突：

```console
docker run -d -e POSTGRES_PASSWORD=secret -p 5432:5432 postgres     # 第一个实例
docker run -d -e POSTGRES_PASSWORD=secret -p 5433:5432 postgres     # 第二个实例（不同宿主机端口）
```

### 2.2 Setting Environment Variables（设置环境变量）

使用 `-e` 设置环境变量：

```console
docker run -e foo=bar postgres env
```

使用 `.env` 文件批量设置环境变量：

```console
docker run --env-file .env postgres env
```

### 2.3 Restricting Container Resources（限制容器资源）

使用 `--memory` 和 `--cpus` 限制 CPU 和内存：

```console
docker run -e POSTGRES_PASSWORD=secret --memory="512m" --cpus="0.5" postgres
```

使用 `docker stats` 监控实时资源使用情况。

### 2.4 Custom Networks（自定义网络）

创建自定义网络以获得更好的隔离和 DNS 解析：

```console
docker network create mynetwork
docker network ls
docker run -d -e POSTGRES_PASSWORD=secret -p 5434:5432 --network mynetwork postgres
```

**默认桥接网络与自定义网络的主要区别**：

| 特性 | 默认桥接网络 | 自定义网络 |
|------|-------------|-----------|
| DNS 解析 | 仅 IP 地址 | 可通过名称/别名解析 |
| 隔离性 | 所有容器互通 | 只有同网络容器可通信 |
| 生产环境 | 不推荐 | 推荐 |

### 2.5 Overriding CMD and ENTRYPOINT（覆盖 CMD 和 ENTRYPOINT）

**使用 Docker Compose**：

```yaml
services:
  postgres:
    image: postgres:18
    entrypoint: ["docker-entrypoint.sh", "postgres"]
    command: ["-h", "localhost", "-p", "5432"]
    environment:
      POSTGRES_PASSWORD: secret
```

**使用 `docker run`**：

```console
docker run -e POSTGRES_PASSWORD=secret postgres docker-entrypoint.sh -h localhost -p 5432
```

## 三、Sharing Local Files with Containers（与容器共享本地文件）

Docker 提供两种存储选项用于在宿主机和容器之间共享文件：**volume** 和 **bind mount**。

### 3.1 Volume 与 Bind Mount 的区别

| 特性 | Volume | Bind Mount |
|------|--------|-------------|
| 用途 | 持久化容器数据 | 共享宿主机文件/目录 |
| 管理方式 | Docker 管理 | 宿主机文件系统 |
| 适用场景 | 数据库、生产数据 | 开发环境、配置文件 |

- **Volume**：适合持久化容器内部生成的数据（参见第四部分）
- **Bind mount**：适合开发环境，直接共享代码和配置文件

### 3.2 使用 Bind Mount

**`-v` 方式**（更简洁）：

```console
docker run -v /HOST/PATH:/CONTAINER/PATH -it nginx
```

如果宿主机位置不存在，`-v` 会自动创建目录。

**`--mount` 方式**（更精细的控制，推荐）：

```console
docker run --mount type=bind,source=/HOST/PATH,target=/CONTAINER/PATH,readonly nginx
```

如果宿主机位置不存在，`--mount` 会生成错误。

### 3.3 文件权限控制

使用 `:ro`（只读）或 `:rw`（读写）标志：

```console
docker run -v HOST-DIRECTORY:/CONTAINER-DIRECTORY:rw nginx
```

- 只读 (`ro`)：容器可读取但不能修改宿主机文件
- 读写 (`rw`)：容器修改会反映到宿主机

> **Synchronized File Share**：对于大型代码库，Docker Desktop 的同步文件共享功能可以利用同步文件系统缓存提升 bind mount 性能。

### 3.4 实践示例

使用 bind mount 将本地网页文件共享到 Apache 容器：

```console
mkdir public_html
# 创建 index.html 文件后...

# -v 方式
docker run -d --name my_site -p 8080:80 -v .:/usr/local/apache2/htdocs/ httpd:2.4

# --mount 方式
docker run -d --name my_site -p 8080:80 --mount type=bind,source=./,target=/usr/local/apache2/htdocs/ httpd:2.4
```

## 四、Persisting Container Data（持久化容器数据）

容器具有临时性：当容器被删除时，其文件更改也会被删除。**Volume** 提供了超越单个容器生命周期的数据持久化能力。

### 4.1 Container Volumes（容器卷）

Volume 可以理解为从容器内部到容器外部的快捷方式或符号链接：

```console
docker volume create log-data
docker run -d -p 80:80 -v log-data:/logs docker/welcome-to-docker
```

如果 volume 不存在，Docker 会自动创建。容器写入 `/logs` 的所有文件都保存在 volume 中，删除容器后数据不会丢失。

**使用 volumes 共享文件**：可以将同一个 volume 挂载到多个容器实现文件共享。

### 4.2 Managing Volumes（管理 Volumes）

```console
docker volume ls                          # 列出所有 volumes
docker volume rm <volume-name-or-id>      # 删除指定 volume
docker volume prune                       # 删除所有未使用的 volumes
```

### 4.3 实践示例：持久化 PostgreSQL 数据

```console
# 启动带 volume 的 Postgres 容器
docker run --name=db -e POSTGRES_PASSWORD=secret -d -v postgres_data:/var/lib/postgresql postgres:18

# 连接数据库
docker exec -ti db psql -U postgres

# 在 psql 中创建数据
CREATE TABLE tasks (
    id SERIAL PRIMARY KEY,
    description VARCHAR(100)
);
INSERT INTO tasks (description) VALUES ('Finish work'), ('Have fun');
SELECT * FROM tasks;

# 退出并删除容器
\q
docker stop db
docker rm db

# 用同一个 volume 启动新容器，数据仍然存在
docker run --name=new-db -d -v postgres_data:/var/lib/postgresql postgres:18
docker exec -ti new-db psql -U postgres -c "SELECT * FROM tasks"
```

## 五、Multi-container Applications（多容器应用）

随着应用增长，使用单个 `docker run` 命令管理多个容器变得困难。**Docker Compose** 通过在单个 `compose.yml` 文件中定义整个多容器应用来解决这个问题。

### 5.1 `docker run` 的局限性

| 问题 | 说明 |
|------|------|
| 命令复杂 | 多个容器需要多条 `docker run` 命令，容易出错 |
| 依赖管理 | 需要手动按特定顺序启动容器并管理网络连接 |
| 扩展困难 | 每个应用需要独立的命令，难以扩展单个服务 |
| 数据管理分散 | 每个容器需单独配置 volume |
| 环境变量繁琐 | 每个容器需单独设置环境变量 |

### 5.2 Docker Compose 的优势

- 在单个 YAML 文件中定义整个多容器应用，集中配置
- 按特定顺序启动容器并管理网络连接
- 可以简单地对单服务进行扩缩容
- 轻松实现持久化 volumes
- 一次性设置所有环境变量

### 5.3 实践示例：多容器 Web 应用

一个包含 Nginx（反向代理）、Node.js Web 应用和 Redis 数据库的多容器设置。

**使用 `docker run` 方式**（繁琐）：

```console
docker network create sample-app

docker run -d --name redis --network sample-app --network-alias redis redis
docker run -d --name web1 -h web1 --network sample-app --network-alias web1 web
docker run -d --name web2 -h web2 --network sample-app --network-alias web2 web
docker run -d --name nginx --network sample-app -p 80:80 nginx

docker ps   # 验证所有容器运行状态
```

**使用 Docker Compose 方式**（简洁）：

```yaml
services:
  redis:
    image: redis
  web1:
    build: ./web
    hostname: web1
  web2:
    build: ./web
    hostname: web2
  nginx:
    build: ./nginx
    ports:
      - 80:80
```

```console
docker compose up -d --build     # 一键启动所有服务
```

## 总结

| 概念 | 关键命令/语法 | 用途 |
|------|-------------|------|
| 发布端口 | `-p HOST:CONTAINER` | 将容器端口暴露给外部访问 |
| 覆盖默认配置 | `-e`, `--memory`, `--cpus`, `--network`, `entrypoint`, `command` | 定制容器运行时行为 |
| 共享本地文件 | `-v` / `--mount type=bind` | 开发时将宿主机文件映射到容器 |
| 持久化数据 | `-v volume_name:/path` | 超越容器生命周期的数据保存 |
| 多容器应用 | `compose.yaml` + `docker compose up` | 定义和管理整个应用栈 |
