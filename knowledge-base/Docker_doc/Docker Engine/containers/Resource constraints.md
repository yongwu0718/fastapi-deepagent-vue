# 资源限制（Resource constraints）

默认情况下，容器没有资源限制，可以使用主机内核调度器允许的任意资源。Docker 提供了通过 `docker run` 命令的运行时配置标志来控制容器可以使用的内存或 CPU 的方法。本节详细说明何时应设置此类限制以及设置它们可能产生的影响。

其中许多功能要求您的内核支持 Linux 能力。您可以使用 [`docker info`](/reference/cli/docker/system/info/) 命令检查支持情况。如果您的内核禁用了某项能力，您可能会在输出的末尾看到类似以下的警告：

```console
WARNING: No swap limit support
```

请查阅操作系统的文档以启用这些功能。另请参阅 [Docker Engine 故障排除指南](/engine/daemon/troubleshoot/#kernel-cgroup-swap-limit-capabilities) 以获取更多信息。

## 内存（Memory）

### 了解内存不足的风险

不要让正在运行的容器消耗过多主机内存，这一点非常重要。在 Linux 主机上，如果内核检测到没有足够的内存来执行重要的系统功能，它会抛出 `OOME`（即内存不足异常），并开始终止进程以释放内存。任何进程都可能被终止，包括 Docker 和其他重要的应用程序。如果终止了错误的进程，这可能会导致整个系统瘫痪。

Docker 尝试通过调整 Docker 守护进程的 OOM 优先级来减轻这些风险，使其比系统上的其他进程更不容易被终止。容器的 OOM 优先级不会被调整。这使得单个容器被终止的可能性高于 Docker 守护进程或其他系统进程。您不应尝试通过手动将守护进程或容器的 `--oom-score-adj` 设置为极端的负数，或在容器上设置 `--oom-kill-disable` 来规避这些保护措施。

有关 Linux 内核 OOM 管理的更多信息，请参阅[内存不足管理（Out of Memory Management）](https://www.kernel.org/doc/gorman/html/understand/understand016.html)。

您可以通过以下方式减轻因 OOME 导致系统不稳定的风险：

- 在生产环境部署之前进行测试，了解应用程序的内存需求。
- 确保您的应用程序仅在具有足够资源的主机上运行。
- 限制容器可以使用的内存量，如下所述。
- 注意 Docker 主机上的交换（swap）配置。交换比内存慢，但可以缓冲系统内存不足的情况。
- 考虑将您的容器转换为[服务（service）](/engine/swarm/services/)，并使用服务级别的约束和节点标签，确保应用程序仅在具有足够内存的主机上运行。

### 限制容器的内存访问

Docker 可以强制设置硬性（hard）或软性（soft）内存限制。

- 硬性限制使容器不能使用超过固定数量的内存。
- 软性限制允许容器使用尽可能多的内存，除非满足某些条件，例如内核检测到主机内存不足或发生争用时。

其中一些选项在单独使用或同时设置多个选项时会产生不同的效果。

大多数选项接受一个正整数，后跟 `b`、`k`、`m`、`g` 后缀，分别表示字节、千字节、兆字节或千兆字节。

| 选项                      | 描述                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| :------------------------ | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `-m` 或 `--memory=`       | 容器可以使用的最大内存量。如果设置此选项，最小允许值为 `6m`（6 兆字节）。也就是说，您必须将值设置为至少 6 兆字节。                                                                                                                                                                                                                                                                                                            |
| `--memory-swap`\*         | 允许此容器交换到磁盘的内存量。请参阅 [`--memory-swap` 详细信息](#--memory-swap-details)。                                                                                                                                                                                                                                                                                                                                                                    |
| `--memory-swappiness`     | 默认情况下，主机内核可以交换掉容器使用的匿名页面的百分比。您可以将 `--memory-swappiness` 设置为 0 到 100 之间的值，以调整此百分比。请参阅 [`--memory-swappiness` 详细信息](#--memory-swappiness-details)。                                                                                                                                                                                                                                                   |
| `--memory-reservation`    | 允许您指定一个小于 `--memory` 的软限制，当 Docker 检测到主机内存不足或争用时，该限制会被激活。如果使用 `--memory-reservation`，必须设置得比 `--memory` 小才能生效。由于是软限制，它不能保证容器不会超过该限制。                                                                                                                                                                                                                                         |
| `--oom-kill-disable`      | 默认情况下，如果发生内存不足（OOM）错误，内核会终止容器中的进程。要更改此行为，请使用 `--oom-kill-disable` 选项。仅在同时设置了 `-m/--memory` 选项的容器上禁用 OOM killer。如果未设置 `-m` 标志，主机可能会耗尽内存，内核可能需要终止主机系统的进程来释放内存。                                                                                                                                                                                              |

有关 cgroup 和内存的更多信息，请参阅[内存资源控制器（Memory Resource Controller）](https://www.kernel.org/doc/Documentation/cgroup-v1/memory.txt) 的文档。

### `--memory-swap` 详细信息

`--memory-swap` 是一个修饰性标志，仅在同时设置了 `--memory` 时才有意义。使用交换允许容器在耗尽其可用的所有 RAM 后将多余的内存需求写入磁盘。对于频繁将内存交换到磁盘的应用程序，会有性能损失。

它的设置可能会产生复杂的效果：

- 如果将 `--memory-swap` 设置为正整数，则必须同时设置 `--memory` 和 `--memory-swap`。`--memory-swap` 表示可以使用的内存和交换的总量，而 `--memory` 控制非交换内存的使用量。因此，如果设置 `--memory="300m"` 和 `--memory-swap="1g"`，容器可以使用 300m 内存和 700m 交换（`1g - 300m`）。

- 如果将 `--memory-swap` 设置为 `0`，则该设置被忽略，视为未设置。

- 如果将 `--memory-swap` 设置为与 `--memory` 相同的值，并且 `--memory` 设置为正整数，则**容器无法访问交换**。请参阅[阻止容器使用交换](#prevent-a-container-from-using-swap)。

- 如果未设置 `--memory-swap`，但设置了 `--memory`，那么如果主机配置了交换内存，容器可以使用与 `--memory` 设置量相同的交换。例如，如果设置 `--memory="300m"` 而未设置 `--memory-swap`，容器总共可以使用 600m 的内存和交换。

- 如果明确将 `--memory-swap` 设置为 `-1`，则允许容器使用无限的交换，直到主机系统可用的上限。

- 在容器内部，`free` 等工具报告的是主机的可用交换，而不是容器内部的可用交换。不要依赖 `free` 或类似工具的输出来判断是否存在交换。

#### 阻止容器使用交换

如果将 `--memory` 和 `--memory-swap` 设置为相同的值，这将阻止容器使用任何交换。这是因为 `--memory-swap` 是可以使用的内存和交换的总量，而 `--memory` 仅仅是可用的物理内存量。

### `--memory-swappiness` 详细信息

- 值为 0 表示关闭匿名页面交换。
- 值为 100 表示将所有匿名页面设置为可交换。
- 默认情况下，如果不设置 `--memory-swappiness`，该值会从主机继承。

## CPU

默认情况下，每个容器对主机 CPU 周期的访问是不受限制的。您可以设置各种约束来限制给定容器对主机 CPU 周期的访问。大多数用户使用并配置[默认 CFS 调度器](#configure-the-default-cfs-scheduler)。您也可以配置[实时调度器](#configure-the-real-time-scheduler)。

### 配置默认 CFS 调度器

CFS 是 Linux 内核用于普通 Linux 进程的 CPU 调度器。有几个运行时标志允许您配置容器对 CPU 资源的访问量。使用这些设置时，Docker 会修改主机上容器 cgroup 的设置。

| 选项                    | 描述                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| :---------------------- | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `--cpus=<value>`        | 指定容器可以使用的可用 CPU 资源量。例如，如果主机有两个 CPU，并且您设置 `--cpus="1.5"`，则容器最多保证使用一个半的 CPU。这等效于设置 `--cpu-period="100000"` 和 `--cpu-quota="150000"`。                                                                                                                                                                                                                                                                                                                                                                                                                |
| `--cpu-period=<value>`  | 指定 CPU CFS 调度器周期，与 `--cpu-quota` 一起使用。默认值为 100000 微秒（100 毫秒）。大多数用户不修改此默认值。对于大多数用例，`--cpus` 是更方便的替代选项。                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| `--cpu-quota=<value>`   | 对容器施加 CPU CFS 配额。在每个 `--cpu-period` 微秒内，容器在被限制之前可以使用的微秒数。因此，它充当有效的上限。对于大多数用例，`--cpus` 是更方便的替代选项。                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| `--cpuset-cpus`         | 限制容器可以使用的特定 CPU 或核心。如果您有多个 CPU，可以使用逗号分隔的列表或连字符分隔的范围来指定。第一个 CPU 编号为 0。有效的值可能是 `0-3`（使用第一、第二、第三和第四个 CPU）或 `1,3`（使用第二个和第四个 CPU）。                                                                                                                                                                                                                                                                                                                                                                                  |
| `--cpu-shares`          | 将此标志设置为大于或小于默认值 1024，以增加或减少容器的权重，使其获得更多或更少比例的主机 CPU 周期。仅当 CPU 周期受限时才会强制执行此设置。当有足够的 CPU 周期可用时，所有容器都会按需使用 CPU。因此，这是一个软限制。`--cpu-shares` 不会阻止容器在 Swarm 模式下被调度。它会在可用的 CPU 周期内优先分配容器的 CPU 资源。它不保证或预留任何特定的 CPU 访问权限。 |

如果您有 1 个 CPU，以下每个命令都保证容器每秒最多使用 50% 的 CPU。

```console
$ docker run -it --cpus=".5" ubuntu /bin/bash
```

这等效于手动指定 `--cpu-period` 和 `--cpu-quota`：

```console
$ docker run -it --cpu-period=100000 --cpu-quota=50000 ubuntu /bin/bash
```

### 配置实时调度器

您可以配置容器使用实时调度器，用于无法使用 CFS 调度器的任务。在[配置 Docker 守护进程](#configure-the-docker-daemon)或[配置单个容器](#configure-individual-containers)之前，您需要[确保主机内核配置正确](#configure-the-host-machines-kernel)。

> [!WARNING]
>
> CPU 调度和优先级是高级内核级功能。大多数用户无需更改这些默认值。错误设置这些值可能导致主机系统不稳定或无法使用。

#### 配置主机内核

通过运行 `zcat /proc/config.gz | grep CONFIG_RT_GROUP_SCHED` 或检查文件 `/sys/fs/cgroup/cpu.rt_runtime_us` 是否存在，来验证 Linux 内核中是否启用了 `CONFIG_RT_GROUP_SCHED`。有关配置内核实时调度器的指导，请查阅操作系统的文档。

#### 配置 Docker 守护进程

要使用实时调度器运行容器，请使用 `--cpu-rt-runtime` 标志启动 Docker 守护进程，该标志设置每个运行时期间为实时任务保留的最大微秒数。例如，在默认周期为 1000000 微秒（1 秒）的情况下，设置 `--cpu-rt-runtime=950000` 确保使用实时调度器的容器在每个 1000000 微秒的周期内可以运行 950000 微秒，为非实时任务留下至少 50000 微秒。在使用 `systemd` 的系统上使此配置永久生效，请为 `docker` 服务创建一个 systemd 单元文件。例如，请参阅有关如何配置守护进程使用代理的说明，使用 [systemd 单元文件](/engine/daemon/proxy/#systemd-unit-file)。

#### 配置单个容器

在通过 `docker run` 启动容器时，您可以传递几个标志来控制容器的 CPU 优先级。有关适当值的详细信息，请查阅操作系统的文档或 `ulimit` 命令。

| 选项                           | 描述                                                                                                                                                                                              |
| :----------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `--cap-add=sys_nice`           | 授予容器 `CAP_SYS_NICE` 能力，允许容器提高进程 `nice` 值、设置实时调度策略、设置 CPU 亲和性以及执行其他操作。                                                                                     |
| `--cpu-rt-runtime=<value>`     | 在 Docker 守护进程的实时调度器周期内，容器可以以实时优先级运行的最大微秒数。您还需要 `--cap-add=sys_nice` 标志。                                                                                   |
| `--ulimit rtprio=<value>`      | 允许容器的最大实时优先级。您还需要 `--cap-add=sys_nice` 标志。                                                                                                                                      |

以下示例命令在 `debian:jessie` 容器上设置了这三个标志。

```console
$ docker run -it \
    --cpu-rt-runtime=950000 \
    --ulimit rtprio=99 \
    --cap-add=sys_nice \
    debian:jessie
```

如果内核或 Docker 守护进程配置不正确，会发生错误。
