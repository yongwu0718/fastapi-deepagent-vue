# 并发与 async / await

关于路径操作函数的 `async def` 语法细节，以及有关异步代码、并发和并行的背景知识。
## 赶时间？

如果你正在使用第三方库，并且该库要求你使用 `await` 来调用它，例如

```python
results = await some_library()
```

那么，请使用 `async def` 来声明你的_路径操作函数_，如下所示

```python
@app.get('/') 
async def read_results():     
	results = await some_library()    
	return results
```

注意

你只能在通过 `async def` 创建的函数内部使用 `await`。

如果你使用的第三方库需要与某些对象（数据库、API、文件系统等）通信，且不支持使用 `await`（目前大多数数据库库都是这种情况），那么请像往常一样，仅使用 `def` 来声明你的_路径操作函数_，如下所示

```python
@app.get('/') 
def results():     
	results = some_library()    
	return results
```

---

如果你的应用程序（无论如何）不需要与任何其他对象通信并等待其响应，请使用 `async def`，即使你不需要在内部使用 `await`。

---

如果你实在拿不准，就使用普通的 `def`。

---

**注意**：你可以在你的 路径操作 函数中根据需要混合使用 `def` 和 `async def`，并为每个函数选择最合适的选项。FastAPI 会正确地处理它们。

无论如何，在上述任何情况下，FastAPI 仍将以异步方式运行，并且速度极快。

但遵循上述步骤，FastAPI 将能够进行一些性能优化。

## 异步代码

异步代码意味着编程语言 💬 有一种方式可以告知计算机/程序 🤖：在代码执行到某个点时，它 🤖 需要等待_其他地方_的_某件东西_完成。假设那个_东西_叫做“慢文件” 📝。

因此，在那段时间里，计算机可以去处理其他工作，而等待“慢文件” 📝 完成。

随后，每当有机会时（比如再次进入等待状态），或者当它 🤖 完成了当时所有手头的工作时，计算机/程序 🤖 就会回来检查。它 🤖 会查看正在等待的任务中是否有已经完成的，并执行相应的后续操作。

接着，它 🤖 获取第一个完成的任务（比如我们的“慢文件” 📝），并继续执行与其相关的逻辑。

那个“等待其他东西”通常指的是相对于处理器和内存速度而言相对“缓慢”的 I/O 操作，例如等待：

- 客户端通过网络发送数据
- 你的程序发送的数据被客户端通过网络接收
- 系统从磁盘读取文件内容并将其交给你的程序
- 你的程序交给系统的内容被写入磁盘
- 远程 API 操作
- 数据库操作完成
- 数据库查询返回结果
- 等等。

由于执行时间主要消耗在等待 I/O 操作上，因此这类操作被称为“I/O 密集型”操作。

之所以称为“异步”，是因为计算机/程序不需要与慢速任务“同步”，即不需要在等待任务完成时什么都不做，完全同步地等待结果以便继续工作。

相反，作为“异步”系统，任务一旦完成，可以排队等候一小会儿（几微秒），待计算机/程序忙完当前手头的活儿，再回来获取结果并继续处理。

对于“同步”（与“异步”相反），人们通常也使用“顺序”一词，因为计算机/程序必须按顺序执行所有步骤，即使这些步骤涉及等待，也不会切换到其他任务。

### 并发与汉堡

上述关于**异步**代码的概念有时也被称为 **"并发 (concurrency)"**。它与 **"并行 (parallelism)"** 是不同的。

**并发**和**并行**都涉及“不同的事情在同一时间段内发生”。

但_并发_和_并行_之间的细节差异很大。

### 并发比并行更好吗？

不！那不是这个故事的寓意。

并发不同于并行。它在**特定**涉及大量等待的场景中更好。因此，对于 Web 应用程序开发，它通常比并行好得多。但并非适用于所有情况。

为了平衡这一点，想象下面这个短故事。

> 你必须打扫一间大而脏的房子。

_没错，就是整个故事。._

---

任何地方都没有等待 🕙，只是有大量的工作需要在房子的多个地方完成。

你可以像汉堡例子那样轮换，先客厅，再厨房，但由于你并没有在等待 🕙 任何东西，只是不断地打扫，轮换并不会产生任何影响。

无论是否轮换（并发），完成工作所需的时间是一样的，你完成的总工作量也一样。

但在这种情况下，如果你能叫来8位以前的收银员/厨师/现在的清洁工，每个人（加上你）负责清理房子的一个区域，你们就可以**并行**地完成所有工作，有了额外的帮助，完成速度会快得多。

在这种情况下，每一位清洁工（包括你）都是一个处理器，负责完成他们那部分工作。

由于大部分执行时间都被实际工作（而不是等待）所占据，而计算机的工作是由 CPU 完成的，所以这些问题被称为“CPU 密集型”。

---

CPU 密集型操作的常见例子是需要复杂数学处理的事情。

例如

- **音频**或**图像处理**。
- **计算机视觉**：图像由数百万个像素组成，每个像素有3个值/颜色，处理这些通常需要同时对这些像素进行计算。
- **机器学习**：通常需要大量的“矩阵”和“向量”乘法。试想一个巨大的数字电子表格，同时将它们全部相乘。
- **深度学习**：这是机器学习的一个子领域，因此同样适用。只是不仅是单一的数字表格，而是海量的表格集，在许多情况下，你需要使用特殊的处理器来构建和/或使用这些模型。

再加上 Python 是**数据科学**、机器学习尤其是深度学习的主要语言这一事实，使得 FastAPI 非常适合数据科学/机器学习 Web API 和应用程序（以及其他许多应用）。

## `async` 与 `await`

现代版本的 Python 提供了一种非常直观的方式来定义异步代码。这使得它看起来就像正常的“顺序”代码，并在合适的时机为你完成“等待”操作。

当某个操作在得出结果前需要等待，且支持这些新的 Python 特性时，你可以这样编写代码：

```python
burgers = await get_burgers(2)
```

这里的关键是 `await`。它告诉 Python，必须先等待 ⏸ `get_burgers(2)` 完成它的工作 🕙，然后才能将结果存储在 `burgers` 中。通过这种方式，Python 会知道在等待期间它可以去执行其他事情 🔀 ⏯（例如接收另一个请求）。

为了使 `await` 生效，它必须位于支持这种异步特性的函数内部。为此，你只需要使用 `async def` 来声明该函数
```python
async def get_burgers(number: int):     
# Do some asynchronous stuff to create the burgers    
	return burgers
```

...而不是 `def`

```python
# This is not asynchronous 
def get_sequential_burgers(number: int):     
# Do some sequential stuff to create the burgers    
	return burgers
```

使用 `async def`，Python 知道在该函数内部必须关注 `await` 表达式，并且它可以“暂停” ⏸ 该函数的执行，去执行其他事情 🔀，然后再回来。

当你想要调用一个 `async def` 函数时，必须“await”它。所以，这样是行不通的

```python
# This won't work, because get_burgers was defined with: async 
def burgers = get_burgers(2)
```

---

因此，如果你使用的库提示你需要用 `await` 来调用它，那么你需要使用 `async def` 来创建使用该库的_路径操作函数_，如下所示

```python
@app.get('/burgers') 
async def read_burgers():     
	burgers = await get_burgers(2)    
	return burgers
```
### 更多技术细节

你可能已经注意到 `await` 只能在定义为 `async def` 的函数内部使用。

但同时，定义为 `async def` 的函数也必须被“await”。因此，`async def` 函数也只能在定义为 `async def` 的函数内部调用。

关于先有鸡还是先有蛋的问题，你怎么调用第一个 `async` 函数呢？

如果你正在使用 **FastAPI**，则不必担心这一点，因为那个“第一个”函数将是你的_路径操作函数_，而 FastAPI 知道如何正确处理它。

但如果你想在没有 FastAPI 的情况下使用 `async` / `await`，你也可以这样做。

### 其他形式的异步代码

这种使用 `async` 和 `await` 的风格在语言中相对较新。

但它使得处理异步代码变得容易得多。

相同的语法（或几乎完全一致）最近也被包含在现代版本的 JavaScript（在浏览器和 NodeJS 中）中。

但在那之前，处理异步代码要复杂和困难得多。

在旧版本的 Python 中，你可能使用过线程或 Gevent。但这些代码理解、调试和思考起来要复杂得多。

在旧版本的 NodeJS / 浏览器 JavaScript 中，你可能使用过“回调”。这导致了“回调地狱”。

## 协程

**协程 (Coroutine)** 只是一个非常时髦的术语，用来描述由 `async def` 函数返回的东西。Python 知道它类似于一个函数，它可以启动并在某个时刻结束，但也可能在内部被暂停 ⏸，只要其中存在 `await`。

但所有这些使用 `async` 和 `await` 编写异步代码的功能，通常被概括为使用“协程”。它类似于 Go 语言的主要特性——“Goroutines”。

### 路径操作函数

当你使用普通的 `def` 而不是 `async def` 声明_路径操作函数_时，它会在一个外部线程池中运行，然后被 await，而不是直接调用（因为那样会阻塞服务器）。

如果你来自另一个不以这种方式工作的异步框架，并且习惯于用纯 `def` 定义琐碎的计算型_路径操作函数_以获得微小的性能提升（约100纳秒），请注意在 **FastAPI** 中，效果可能恰恰相反。在这些情况下，最好使用 `async def`，除非你的_路径操作函数_包含执行阻塞 I/O 的代码。

尽管如此，在这两种情况下，**FastAPI** 都有可能 仍然比 你之前的框架更快（或者至少相当）。

### 依赖

这同样适用于 依赖项。如果依赖项是标准的 `def` 函数而不是 `async def`，它将在外部线程池中运行。

### 子依赖项

你可以拥有多个互相依赖的 子依赖项（作为函数定义的参数），其中一些可能使用 `async def` 创建，一些使用普通 `def`。这仍然可以工作，且使用普通 `def` 创建的依赖项将在外部线程（来自线程池）中调用，而不是被“await”。

### 其他工具函数

你直接调用的任何其他工具函数都可以使用普通的 `def` 或 `async def` 创建，FastAPI 不会影响你的调用方式。

这与 FastAPI 为你调用的函数形成了对比：_路径操作函数_和依赖项。

如果你的工具函数是一个普通的 `def` 函数，它将直接被调用（正如你在代码中写的那样），而不是在线程池中；如果函数是使用 `async def` 创建的，那么当你调用它时，应该对该函数使用 `await`。
