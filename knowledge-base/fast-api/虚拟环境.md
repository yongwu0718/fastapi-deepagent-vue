# 虚拟环境

在进行 Python 项目开发时，建议使用**虚拟环境**（或类似机制）来隔离每个项目所安装的包。

注意

如果你已经了解虚拟环境，并知道如何创建和使用它们，可以跳过本节。🤓

提示

**虚拟环境**与**环境变量**是不同的。

**环境变量**是系统中可供程序使用的变量。

**虚拟环境**是一个包含某些文件的目录。

注意

本页将教你如何使用**虚拟环境**以及它们的工作原理。

如果你准备采用一种**为你管理一切**（包括安装 Python）的工具，可以尝试 uv。

当你**首次**开始开发 Python 项目时，请在**你的项目目录内**创建一个虚拟环境。

提示

## 添加 `.gitignore`

如果你正在使用 **Git**（你应该用），请添加一个 `.gitignore` 文件，以将 `.venv` 中的所有内容从 Git 中排除。

提示

如果你使用 `uv` 创建了虚拟环境，它已经为你完成了此操作，你可以跳过这一步。😎

提示

在创建虚拟环境后，**执行一次**即可。

echo " * " > .venv/.gitignore  

命令含义

- `echo "*"`: 将在终端中“打印”文本 `*`（下一部分会对此稍作修改）
- `>`: 表示将左侧命令打印到终端的内容重定向写入到右侧的文件中，而不是显示在屏幕上
- `.gitignore`: 要写入文本的文件名

对于 Git 而言，`*` 表示“所有内容”。因此，它将忽略 `.venv` 目录中的所有内容。

该命令将创建一个内容为以下内容的 `.gitignore` 文件

## 为什么要使用虚拟环境

要使用 FastAPI，你需要安装 Python。

之后，你还需要**安装** FastAPI 和你想要使用的任何其他**包**。

安装包时，通常会使用随 Python 一起提供的 `pip` 命令（或类似的替代方案）。

然而，如果你直接使用 `pip`，包会被安装在你的**全局 Python 环境**（即 Python 的全局安装位置）中。

### 问题所在

那么，在全局 Python 环境中安装包有什么问题呢？

在某个阶段，你可能会编写许多依赖于**不同包**的不同程序。而且你所处理的某些项目会依赖于同一个包的**不同版本**。😱

例如，你可以创建一个名为 `philosophers-stone` 的项目，该程序依赖于另一个名为 **`harry` 的包，版本为 `1`**。因此，你需要安装 `harry`。

然后，在以后的某个时候，你创建了另一个名为 `prisoner-of-azkaban` 的项目，该项目也依赖于 `harry`，但需要 **`harry` 的版本 `3`**。

现在的问题是，如果你将包安装在全局（全局环境）而不是本地的**虚拟环境**中，你将不得不选择安装哪个版本的 `harry`。

如果你想运行 `philosophers-stone`，你将需要安装 `harry` 版本 `1`，例如使用

pip install "harry==1"  
restart ↻

最终你的全局 Python 环境中就会安装 `harry` 版本 `1`。

但如果你之后想运行 `prisoner-of-azkaban`，你需要卸载 `harry` 版本 `1` 并安装 `harry` 版本 `3`（或者安装版本 `3` 会自动卸载版本 `1`）。

pip install "harry==3"  
restart ↻

最终你的全局 Python 环境中就会安装 `harry` 版本 `3`。

此时如果你尝试再次运行 `philosophers-stone`，它很有可能**无法工作**，因为它需要的是 `harry` 版本 `1`。

提示

Python 包通常会尽力**避免在新版本中引入破坏性更改**，但最好还是保持安全，在有把握运行测试以确保一切正常的情况下，有意识地安装新版本。

现在，想象一下所有项目依赖的**许多**其他**包**。这非常难以管理。你很可能会在运行某些项目时因为使用了某些**不兼容的包版本**而报错，却不知道问题出在哪里。

此外，根据你的操作系统（如 Linux、Windows、macOS），它可能已经预装了 Python。在这种情况下，它可能预装了一些**系统所需**的特定版本的包。如果你在全局 Python 环境中安装包，最终可能会**破坏**某些随操作系统一起提供的程序。

## 包安装在哪里

当你安装 Python 时，它会在你的计算机上创建一些包含文件的目录。

其中一些目录负责存放你安装的所有包。

当你运行

fast →💬 Don't run this now, it's just an example 🤓pip install "fastapi[standard]"████ 10%

它会从 PyPI 下载一个包含 FastAPI 代码的压缩文件。

它还会**下载** FastAPI 依赖的其他包的文件。

然后它会**解压**所有这些文件，并将它们放入你计算机上的一个目录中。

默认情况下，它会将这些下载并解压的文件放入随 Python 安装一起提供的目录中，这就是**全局环境**。

## 什么是虚拟环境

解决将所有包放在全局环境中的问题的方案是：为你工作的每个项目使用一个**虚拟环境**。

虚拟环境是一个**目录**，与全局环境非常相似，你可以在其中为某个项目安装包。

这样，每个项目都有自己的虚拟环境（`.venv` 目录）和各自的包。

## 激活虚拟环境意味着什么

当你激活虚拟环境时，例如使用

Linux, macOSWindows PowerShellWindows Bash

.venv\Scripts\Activate.ps1  
restart ↻

该命令将创建或修改一些可供后续命令使用的环境变量。

其中一个变量是 `PATH` 变量。

提示

你可以在 环境变量 部分详细了解 `PATH` 环境变量。

激活虚拟环境会将虚拟环境的路径（Linux 和 macOS 上为 `.venv/bin`，Windows 上为 `.venv\Scripts`）添加到 `PATH` 环境变量中。

假设在激活环境之前，`PATH` 变量看起来像这样

Linux, macOSWindows

`C:\Windows\System32`

这意味着系统会在以下位置查找程序

- `C:\Windows\System32`

激活虚拟环境后，`PATH` 变量看起来会像这样

Linux, macOSWindows

`C:\Users\user\code\awesome-project\.venv\Scripts;C:\Windows\System32`

这意味着系统现在会首先开始在以下位置查找程序

`C:\Users\user\code\awesome-project\.venv\Scripts`

然后再查看其他目录。

所以，当你在终端中输入 `python` 时，系统会找到位于以下位置的 Python 程序

`C:\Users\user\code\awesome-project\.venv\Scripts\python`

并使用它。

一个重要的细节是，它会将虚拟环境路径放在 `PATH` 变量的**开头**。系统会**在**找到任何其他可用 Python 之前先找到它。这样，当你运行 `python` 时，它将使用**来自虚拟环境的** Python，而不是任何其他 `python`（例如，来自全局环境的 `python`）。

激活虚拟环境还会改变其他一些事项，但这是它所做的最重要的事情之一。

## 检查虚拟环境

当你检查虚拟环境是否激活时，例如使用

Linux, macOS, Windows BashWindows PowerShell

Get-Command python  
C:\Users\user\code\awesome-project\.venv\Scripts\python  
  
restart ↻

这意味着将使用的 `python` 程序正是**虚拟环境中的那一个**。

在 Linux 和 macOS 上，你使用 `which`；在 Windows PowerShell 上，你使用 `Get-Command`。

该命令的工作方式是检查 `PATH` 环境变量，按**顺序**遍历每个路径，查找名为 `python` 的程序。一旦找到，它就会**显示该程序的路径**。

最重要的一点是，当你调用 `python` 时，执行的正是那个路径下的“`python`”。

因此，你可以确认你是否处于正确的虚拟环境中。

提示

很容易出现这样的情况：激活了一个虚拟环境，获取了一个 Python 版本，然后**转到了另一个项目**。

第二个项目**将无法工作**，因为你正在使用**错误的 Python**，它是来自另一个项目的虚拟环境的。

能够检查正在使用哪个 `python` 是非常有用的。🤓

然后你想处理**另一个项目** `prisoner-of-azkaban`。

你进入该项目

如果你不为 `philosophers-stone` 停用虚拟环境，当你在终端中运行 `python` 时，它仍会尝试使用 `philosophers-stone` 的 Python。

fast →cd ~/code/prisoner-of-azkaban  

## 替代方案

这是一个帮助你入门并教你**底层**工作原理的简单指南。

有很多**替代方案**可以用来管理虚拟环境、包依赖（requirements）和项目。

一旦你准备好使用一个工具来**管理整个项目**、包依赖、虚拟环境等，我建议你尝试 uv。

`uv` 可以做很多事情，它可以

- 为你**安装 Python**，包括不同版本
- 管理项目的**虚拟环境**
- 安装**包**
- 管理项目的**包依赖和版本**
- 确保你拥有一组**精确的**包及其版本可以安装（包括它们的依赖项），这样你就可以确保在生产环境运行项目时与开发时在电脑上的环境完全一致，这被称为**锁定（locking）**
- 以及许多其他事情
