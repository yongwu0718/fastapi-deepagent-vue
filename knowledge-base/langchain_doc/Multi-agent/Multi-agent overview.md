# Multi-agent

多智能体系统协调专门的组件来处理复杂的工作流。然而，并非每个复杂任务都需要这种方法——一个具有正确（有时是动态的）工具和提示的单个 agent 通常也能达到类似的结果。

对于内置的多智能体支持，请使用 Deep Agents：一个基于 LangChain 构建的更高级别的工具包，内置了子代理、技能、规划、虚拟文件系统和上下文管理。

## 为什么需要多智能体？

当开发人员说他们需要“多智能体”时，他们通常是在寻求以下一种或多种能力：

*  **上下文管理**：提供专门的知识，而不会使模型的上下文窗口过载。如果上下文是无限的且延迟为零，您可以将所有知识转储到一个提示中——但事实并非如此，因此您需要模式来有选择地呈现相关信息。
*  **分布式开发**：允许不同的团队独立开发和维护能力，将它们组合成一个具有清晰边界的更大系统。
*  **并行化**：为子任务生成专门的 worker 并并发执行它们以获得更快的速度。

当单个 agent 拥有太多工具且在决定使用哪些工具时做出糟糕的决策时，当任务需要具有广泛上下文（长提示和领域特定工具）的专业知识时，或者当您需要在满足某些条件后才解锁能力的顺序约束时，多智能体模式尤其有价值。

多智能体设计的核心是**上下文工程**——决定每个 agent 看到什么信息。您系统的质量取决于确保每个 agent 都有权访问其任务所需的数据。

## 模式

以下是构建多智能体系统的主要模式，每种模式适用于不同的用例：

| 模式                                                                  | 工作原理                                                                                                                                                                                        |
| ------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Subagents**             | 主 agent 将子代理作为工具进行协调。所有路由都通过主 agent，它决定何时以及如何调用每个子代理。                                                         |
| **Handoffs**               | 行为根据状态动态变化。工具调用更新触发路由或配置更改的状态变量，从而切换代理或调整当前代理的工具和提示。 |
| **Skills**                   | 按需加载的专业提示和知识。一个 agent 保持控制权，同时根据需要从技能加载上下文。                                                                    |
| **Router**                   | 路由步骤对输入进行分类，并将其导向一个或多个专门的 agent。结果被综合成一个组合响应。                                                                 |
| **Custom workflow** | 使用 LangGraph 构建特定的执行流，混合确定性逻辑和代理行为。将其他模式作为工作流中的节点嵌入。                    |

### 选择模式

使用此表格将您的需求与正确的模式匹配：

| 模式                                                      | 分布式开发 | 并行化 | 多跳 | 直接用户交互 |
  | ------------------------------------------------------------ | :---------------------: | :-------------: | :-------: | :---------------------: |
  | **Subagents** |          ⭐⭐⭐⭐⭐          |      ⭐⭐⭐⭐⭐      |   ⭐⭐⭐⭐⭐   |            ⭐            |
  | **Handoffs**   |            -            |        -        |   ⭐⭐⭐⭐⭐   |          ⭐⭐⭐⭐⭐          |
  | **Skills**       |          ⭐⭐⭐⭐⭐          |       ⭐⭐⭐       |   ⭐⭐⭐⭐⭐   |          ⭐⭐⭐⭐⭐          |
  | **Router**       |           ⭐⭐⭐           |      ⭐⭐⭐⭐⭐      |     -     |           ⭐⭐⭐           |

* **分布式开发**：不同的团队能否独立维护组件？
* **并行化**：多个 agents 能否并发执行？
* **多跳**：该模式是否支持串联调用多个子代理？
* **直接用户交互**：子代理能否直接与用户对话？

您可以混合使用模式！例如，一个 **subagents** 架构可以调用调用自定义工作流或路由器 agents 的工具。Subagents 甚至可以使用 **skills** 模式按需加载上下文。可能性是无穷的！

### 可视化概览

主 agent 将子代理作为工具进行协调。所有路由都通过主 agent。

Agents 通过工具调用相互传递控制权。每个 agent 可以移交给其他 agent 或直接响应用户。

一个 agent 保持控制权，同时按需加载专业的提示和知识。

路由步骤对输入进行分类，并将其导向专门的 agents。结果被综合。

使用 LangSmith 追踪跨 agents 的完整协调流程。按照跟踪快速入门进行设置。

## 性能比较

不同的模式具有不同的性能特征。了解这些权衡有助于您根据延迟和成本要求选择正确的模式。

**关键指标：**

* **模型调用次数**：LLM 调用的数量。调用越多 = 延迟越高（尤其是顺序调用时）和每次请求的 API 成本越高。
* **处理的 Tokens**：所有调用中使用的总上下文窗口大小。Tokens 越多 = 处理成本越高，并可能触及上下文限制。

### 重复请求

> **第 1 轮：** "买咖啡"
> **第 2 轮：** "再买一次咖啡"

用户在同一个对话中重复相同的请求。

| 模式                                                      | 第 2 轮调用次数 | 总计（两轮） | 最适合 |
  | ------------------------------------------------------------ | :----------: | :----------------: | :------: |
  | **Subagents** |       4      |          8         |          |
  | **Handoffs**   |       2      |          5         |     ✅    |
  | **Skills**       |       2      |          5         |     ✅    |
  | **Router**       |       3      |          6         |          |

**再次 4 次调用 → 总计 8 次**

    * Subagents 设计上是**无状态**的——每次调用都遵循相同的流程
    * 主 agent 维护对话上下文，但子代理每次都是全新启动
    * 这提供了强上下文隔离，但重复了完整的流程

**2 次调用 → 总计 5 次**

    * 咖啡 agent 从第 1 轮开始**仍然活跃**（状态持续存在）
    * 不需要交接——agent 直接调用 `buy_coffee` 工具（调用 1）
    * agent 响应用户（调用 2）
    * **通过跳过交接节省了 1 次调用**

**2 次调用 → 总计 5 次**

    * 技能上下文**已加载**到对话历史中
    * 无需重新加载——agent 直接调用 `buy_coffee` 工具（调用 1）
    * agent 响应用户（调用 2）
    * **通过重用已加载的技能节省了 1 次调用**

**再次 3 次调用 → 总计 6 次**

    * Routers 是**无状态**的——每个请求都需要一个 LLM 路由调用
    * 第 2 轮：路由 LLM 调用 (1) → Milk agent 调用 buy\_coffee (2) → Milk agent 响应 (3)
    * 可以通过将其包装在状态 agent 中作为工具来优化

**关键见解：** 有状态模式（Handoffs, Skills）在重复请求上节省了 40-50% 的调用次数。Subagents 保持每次请求的成本一致——这种无状态设计提供了强上下文隔离，但代价是重复的模型调用。

### 多领域

> **用户：** "比较 Python、JavaScript 和 Rust 用于 Web 开发"

每个语言 agent/skill 包含约 2000 个 tokens 的文档。所有模式都可以进行并行工具调用。

| 模式                                                      | 模型调用次数 | 总 Tokens | 最适合 |
| ------------------------------------------------------------ | :---------: | :----------: | :------: |
| **Subagents** |      5      |     \~9K     |     ✅    |
| **Handoffs**   |      7+     |    \~14K+    |          |
| **Skills**       |      3      |     \~15K    |          |
| **Router**       |      5      |     \~9K     |     ✅    |

**5 次调用，\~9K tokens**
![alt text](image-5.png)
每个子代理在**隔离**环境中工作，只处理其相关的上下文。总计：**9K tokens**。

**7+ 次调用，\~14K+ tokens**
![alt text](image-4.png)
Handoffs 按**顺序执行**——无法并行研究所有三种语言。不断增长的对话历史增加了开销。总计：**\~14K+ tokens**。

**3 次调用，\~15K tokens**
![alt text](image-6.png)
加载后，**每次后续调用都会处理全部 6K tokens 的技能文档**。由于上下文隔离，Subagents 总体处理的 tokens 减少了 67%。总计：**15K tokens**。

**5 次调用，\~9K tokens**
![alt text](image-7.png)
Router 使用 **LLM 进行路由**，然后并行调用 agents。与 Subagents 类似，但具有显式的路由步骤。总计：**9K tokens**。

**关键见解：** 对于多领域任务，具有并行执行能力的模式（Subagents, Router）最为高效。Skills 调用次数较少，但由于上下文累积，token 使用量较高。Handoffs 在这里效率低下——它必须顺序执行，并且无法利用并行工具调用来同时咨询多个领域。

### 总结

以下是所有三种场景下模式的比较：

| 模式                                                      | 单次请求 | 重复请求 |     多领域     |
  | ------------------------------------------------------------ | :------: | :------------: | :-------------------: |
  | **Subagents** |  4 calls |  8 calls (4+4) |   5 calls, 9K tokens  |
  | **Handoffs**   |  3 calls |  5 calls (3+2) | 7+ calls, 14K+ tokens |
