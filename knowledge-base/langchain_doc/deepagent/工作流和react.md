## 表格 1：工作流适合的场景及原因

| 场景 | 为什么工作流比 ReAct 更合适 |
| :--- | :--- |
| 流程固定、可预测 | 图结构锁定了执行路径，不会因为 LLM "跑偏"而偏离预期 |
| 需要分阶段控制 | 如你的设计：先工具调用阶段，再纯回答阶段，各阶段职责清晰 |
| 需要确定性输出 | `llm_response` 节点不绑工具，保证最后一步一定是生成回答而非继续调工具 |
| 可审计、可调试 | 每个节点职责单一，出问题时能定位到具体阶段 |
| 对工具调用次数有约束 | 工作流的边可以硬性限制循环次数或路径，ReAct 的循环是 LLM 主导的 |
| 多角色协作 | 不同节点可以用不同 prompt/模型/工具，比如"收集节点"和"总结节点"分工 |

## 表格 2：工作流 vs ReAct 全面对比

| 对比维度 | 工作流 | ReAct |
| :--- | :--- | :--- |
| 决策方式 | 图结构预定义 | LLM 自主每步决策 |
| 适合场景 | 流程已知、需控制 | 探索性、开放式任务 |
| 风险 | 灵活性低 | 可能无限循环、跑偏 |
| 你的 billing 场景 | ✅ 流程固定：选脚本→执行→解读 | ❌ 大材小用且不可控 |
